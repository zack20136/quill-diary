# 測試手則

這份手則用來維持目前的測試分層與命名一致，避免 `test/` 再度回到散落、重疊、難維護的狀態。

## 分層原則

- `core` 只放最值得長期維護的核心規則：`config`、`crypto`、`database`、`storage`、`vault`。
- `helpers` 是測試支援層，依職責拆到 `shared`、`session`、`storage`、`vault`、`features/*`。
- `smoke` 只保留少量 app / features 的煙霧案例，不承擔核心規則驗證。
- 若某個測試同時驗證多個職責，優先下放到更底層的 `core` 或拆成更小的測試。

## 命名原則

- 測試名稱以繁體中文為主，直接描述行為與結果。
- 名稱要能看出場景、條件、預期，例如：
  - `正確復原金鑰可通過驗證`
  - `鎖定且已連線時會停用帳號操作`
  - `修復會將放錯路徑的日記搬回 canonical`
- 避免含糊字眼，例如「正常」、「應該可以」、「測試一個功能」。

## 檔案與目錄

- 同一 domain 的測試盡量放在同一目錄底下。
- `helpers` 不要繼續堆在根目錄，新增 helper 時先想清楚它屬於哪個 domain。
- 煙霧測試統一放在 `test/smoke/`，不要再把 UI / controller 測試散回各 domain。

## 執行方式

- 新增或搬移測試後，至少執行對應 runner 一次，確認沒有 import 斷裂或命名誤植。

## 寫測試前先檢查

1. 這個測試是否已被其他更底層測試覆蓋。
2. 這個行為屬於 `core`、`helpers`、`storage`、`vault`、`session`，還是 `smoke`。
3. 測試名稱是否直接說出條件與預期。
4. 是否使用現有的 helper，而不是再造一份新的 fake。
5. 是否真的需要新增測試；如果只是重複驗證同一條規則，通常該刪或合併。

## 維護提醒

- 先整理結構，再處理重疊案例。
- 繁體中文字串要保留原文，不要轉成 escape 或亂改 encoding。
