# QuillLockDiary 更詳細的實作細節

> 建立日期：2026-05-13  
> 目的：補強《專案計劃書.md》，把產品構想整理成可實作、可拆工、可驗收的工程規格。

---

## 1. 先補強原計劃書的幾個關鍵決策

原計劃書方向正確，但要真正進入開發，建議先把以下幾點固定下來：

### 1.1 明確定義單一真實資料來源

建議固定：

```text
加密後的 entries/*.md.enc 與 assets/*.enc
才是真實資料來源。

SQLite index、manifest、backup manifest
都屬於可重建的衍生資料。
```

這個決策很重要，因為它直接決定：

- index 壞掉能不能救
- 備份是否一定要帶 index
- 匯入與還原時應以什麼資料為準

### 1.2 移除 vaultKey 與 fileKey 的模糊地帶

原計劃書在金鑰層級中寫了 `vaultKey 或 fileKey`，實作時不建議保留模糊空間。

MVP 建議固定：

```text
每個檔案一把 fileKey
不引入可解開整個 vault 的單一 vaultKey
```

原因：

- 單篇救援更自然
- 檔案損毀隔離性更好
- Recovery Key 輪替只需重包裝各檔案 fileKey
- 避免單一 vaultKey 洩漏就整包失守

### 1.3 定義「寫入一致性」而不是只定義格式

目前計劃書描述了檔案格式，但尚未定義寫入策略。建議固定：

```text
所有檔案更新都採 temporary file + fsync + rename replace
索引更新晚於加密檔寫入
只要加密檔存在，index 就可重建
```

這能避免：

- App 當掉時留下半寫入檔案
- index 已更新但加密檔沒寫成功
- 備份抓到不一致狀態

### 1.4 定義備份是一個快照，不是同步

建議補充：

```text
.jbackup 是某一時間點的不可變快照
不是雙向同步資料集
```

這會影響：

- 命名策略
- 還原覆蓋規則
- 多份備份保留策略
- 使用者認知

### 1.5 定義威脅模型與不處理範圍

原計劃書強調安全，但還缺一段明確界線。建議補上：

本 App 要防的風險：

- 手機遺失後，未解鎖者直接讀取 App 檔案
- Google Drive 備份外洩後直接看到明文
- Android Keystore 失效後仍可用 Recovery Key 救援

本 App 不承諾防禦：

- 已解鎖裝置上的惡意鍵盤或螢幕錄影
- 被 root 且執行中遭植入惡意程式的裝置
- 使用者自己外洩 Recovery Key

## 2. 實作總原則

### 2.1 核心資料模型

```text
DiaryEntry
  代表一篇日記的業務資料

EncryptedDocument
  代表一個加密檔案的實體格式

AssetAttachment
  代表圖片或附件

IndexRecord
  代表 SQLite 中可查詢資料
```

### 2.2 模組分層

建議固定為四層：

```text
UI / Feature Layer
  頁面、互動、狀態管理

Application Layer
  use case、流程編排、交易邏輯

Domain Layer
  entity、value object、規則

Infrastructure Layer
  檔案系統、SQLite、Keystore、Google Drive、加密實作
```

避免讓頁面直接操作：

- SQLite
- 檔案路徑
- 加密 header
- Google Drive API

### 2.3 單向依賴

```text
UI -> Application -> Domain -> Infrastructure interface
Infrastructure -> concrete implementation
```

不要讓：

- UI 直接呼叫 `sqflite`
- UI 直接決定 `.md.enc` 路徑
- Feature page 自己拼 backup 檔

## 3. 建議的實體與欄位定義

## 3.1 DiaryEntry

```text
id: String
title: String
date: LocalDate
createdAt: DateTime
updatedAt: DateTime
tags: List<String>
mood: String?
markdownBody: String
attachmentIds: List<String>
isDeleted: bool
```

說明：

- `date` 是日記所屬日期，用於日曆與時間線。
- `createdAt`、`updatedAt` 是技術欄位。
- `markdownBody` 不含 front matter。
- `title` 可為空，但 UI 要決定空標題顯示規則。

## 3.2 AssetAttachment

```text
id: String
entryId: String
mimeType: String
originalFilename: String?
safeFilename: String
width: int?
height: int?
byteSize: int
createdAt: DateTime
sha256: String
```

說明：

- `originalFilename` 不寫入匯出 Markdown。
- `safeFilename` 是 vault 與匯出檔案名。
- `sha256` 用於重複附件判定與完整性驗證。

## 3.3 RecoveryMetadata

```text
vaultId: String
recoveryEnabled: bool
recoveryKeyVersion: int
recoveryKeyHint: String
createdAt: DateTime
kdfAlgorithm: String
```

## 4. ID 與路徑規則

## 4.1 ID 規則

建議全系統固定使用可排序且低碰撞的 ID，例如 ULID。

```text
vault_id: vlt_<ulid>
entry_id: jrn_<ulid>
asset_id: att_<ulid>
backup_id: bkp_<ulid>
device_slot_id: dev_<ulid>
```

理由：

- 比時間戳加亂碼更一致
- 可排序
- 匯入外部資料時較不易撞名

## 4.2 路徑規則

建議使用 entry 的 `date` 切目錄，而不是 `createdAt`。

```text
vault/
  entries/YYYY/MM/<entry_id>.md.enc
  assets/YYYY/MM/<asset_id>.<ext>.enc
```

好處：

- 日曆與匯出路徑直覺一致
- 使用者改日期後可以決定是否搬動檔案

### 4.3 日期變更策略

若使用者把日記日期從 `2026-05-13` 改成 `2026-05-20`，MVP 建議：

```text
更新 index 與 front matter
實體檔案路徑不搬動
```

原因：

- 避免 rename 帶來額外失敗點
- 路徑與日期不完全一致是可接受的內部實作細節

匯出時再依 `date` 產生對外目錄即可。

## 5. 加密格式實作細節

## 5.1 演算法選擇

MVP 建議固定：

```text
content cipher: XChaCha20-Poly1305
key wrap: AES-GCM 或 ChaCha20-Poly1305 包裝 fileKey
KDF: Argon2id
hash: SHA-256
```

若 Flutter 生態在 XChaCha20-Poly1305 支援不足，退而求其次：

```text
content cipher: AES-256-GCM
```

但必須把實作封裝在 `CryptoService`，不要讓上層知道底層差異。

## 5.2 AAD 設計

建議把下列欄位組成 canonical JSON 當作 AAD：

```json
{
  "schema_version": 1,
  "file_id": "jrn_...",
  "vault_id": "vlt_...",
  "content_type": "text/markdown"
}
```

目的：

- 防止 ciphertext 被移花接木到其他檔案
- 驗證 header 與內容綁定

## 5.3 檔案格式固定結構

建議正式定義：

```text
bytes 0..3   : magic = "LDJ1"
bytes 4..7   : header_length (uint32 big-endian)
bytes 8..n   : UTF-8 JSON header
bytes n+1..  : ciphertext
```

額外規定：

- header 一律 UTF-8
- JSON key 順序固定輸出
- header 最大長度限制 64 KB
- 超過限制視為格式錯誤

## 5.4 Key slot 結構

建議把 key slot 抽象成：

```json
{
  "slot_id": "rks_01",
  "type": "recovery",
  "wrap_algorithm": "aes-256-gcm",
  "wrapped_key": "base64...",
  "nonce": "base64...",
  "kdf": {
    "name": "argon2id",
    "salt": "base64...",
    "memory_cost_kib": 65536,
    "time_cost": 3,
    "parallelism": 1
  }
}
```

`device` slot 可沒有 `kdf`，但欄位結構應一致。

## 5.5 Recovery Key 輪替策略

更換 Recovery Key 時不要重寫 plaintext。

流程固定：

```text
1. 逐檔讀 header
2. 用目前可用 slot 解出 fileKey
3. 產生新的 recovery slot
4. 原子性覆蓋 header + ciphertext 檔案
5. 全部成功後才更新 recovery.json version
```

若中途中斷：

- 已完成的檔案有新 slot
- 未完成的檔案保留舊 slot
- 直到全量完成前，不刪除舊 slot

也就是說輪替流程需要支援「雙 recovery slot 暫存期」。

## 6. 寫入、刪除、重建策略

## 6.1 新增或更新日記

建議交易順序：

```text
1. 組出 DiaryEntry domain model
2. 轉成 plaintext markdown
3. 產生或重用 fileKey
4. 輸出 .md.enc.tmp
5. fsync tmp
6. rename replace 成正式檔
7. 更新 SQLite index
8. 更新 manifest 快取
```

如果第 7 步失敗：

- 視為資料仍然安全存在
- 下次啟動可重建 index

## 6.2 刪除日記

MVP 不建議立即實體刪除。

建議：

```text
先做軟刪除 is_deleted = 1
並把 entry 從主要列表隱藏
```

原因：

- 降低誤刪風險
- 還原與同步衝突較少

第二階段再加：

- 垃圾桶
- 永久刪除
- 清除未使用附件

## 6.3 index 重建

提供 `RebuildIndexUseCase`：

```text
掃描 entries/*.md.enc
逐檔解 header
必要時解密 markdown
重建 entries_index、entry_tags、entry_attachments、fts
```

重建模式分兩種：

```text
fast rebuild:
  只讀 header 與必要 metadata

full rebuild:
  解密每篇 markdown 並重建全文搜尋
```

## 8. SQLite 設計補強

SQLite 要被當成完整的查詢層來設計，而不是零散快取。

固定原則：

```text
entries/*.md.enc 與 assets/*.enc 才是真實資料
SQLite 是查詢加速層
SQLite 可整包重建
FTS 可單獨重建
```

## 8.1 建議正式資料表

### entries_index

```sql
CREATE TABLE entries_index (
  id TEXT PRIMARY KEY,
  vault_id TEXT NOT NULL,
  file_path TEXT NOT NULL,
  title TEXT,
  title_normalized TEXT,
  preview_text TEXT,
  date TEXT NOT NULL,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  timezone TEXT,
  mood TEXT,
  weather TEXT,
  location_text TEXT,
  word_count INTEGER NOT NULL DEFAULT 0,
  char_count INTEGER NOT NULL DEFAULT 0,
  attachment_count INTEGER NOT NULL DEFAULT 0,
  has_attachments INTEGER NOT NULL DEFAULT 0,
  is_favorite INTEGER NOT NULL DEFAULT 0,
  is_deleted INTEGER NOT NULL DEFAULT 0,
  deleted_at TEXT,
  schema_version INTEGER NOT NULL DEFAULT 1,
  encrypted_file_size INTEGER,
  encrypted_file_mtime TEXT,
  content_hash TEXT
);
```

欄位用途：

- `title_normalized`：搜尋、排序、去除大小寫與空白差異。
- `preview_text`：首頁、時間線、搜尋結果摘要。
- `timezone`：保留原始建立時區，避免跨時區日期顯示錯亂。
- `weather`、`location_text`：先保留資料欄位，不代表第一版一定要做 UI。
- `char_count`：比 `word_count` 更適合中文。
- `is_favorite`：之後做釘選或收藏不用改 schema。
- `content_hash`：判斷 plaintext 是否變更、協助 rebuild 與除錯。

### entry_tags

```sql
CREATE TABLE entry_tags (
  entry_id TEXT NOT NULL,
  tag TEXT NOT NULL,
  tag_normalized TEXT NOT NULL,
  PRIMARY KEY (entry_id, tag_normalized)
);
```

### entry_attachments

```sql
CREATE TABLE entry_attachments (
  id TEXT PRIMARY KEY,
  entry_id TEXT NOT NULL,
  file_path TEXT NOT NULL,
  mime_type TEXT NOT NULL,
  safe_filename TEXT NOT NULL,
  width INTEGER,
  height INTEGER,
  byte_size INTEGER NOT NULL,
  sha256 TEXT NOT NULL,
  created_at TEXT NOT NULL,
  is_deleted INTEGER NOT NULL DEFAULT 0
);
```

### entries_fts

```sql
CREATE VIRTUAL TABLE entries_fts USING fts5(
  entry_id,
  title,
  body,
  tags,
  preview_text
);
```

### app_kv

```sql
CREATE TABLE app_kv (
  key TEXT PRIMARY KEY,
  value TEXT NOT NULL,
  updated_at TEXT NOT NULL
);
```

建議存放：

- 最後成功備份時間
- index schema version
- 最近一次 rebuild 完成時間
- 最近一次 Recovery 驗證時間

### backup_history

```sql
CREATE TABLE backup_history (
  backup_id TEXT PRIMARY KEY,
  provider TEXT NOT NULL,
  remote_file_id TEXT,
  created_at TEXT NOT NULL,
  status TEXT NOT NULL,
  entry_count INTEGER NOT NULL,
  asset_count INTEGER NOT NULL,
  byte_size INTEGER,
  error_code TEXT
);
```

## 8.2 FTS 實作策略

建議不要把 `entries_fts` 當唯一真相來源。

流程：

```text
儲存日記
-> 更新 entries_index
-> 更新 entry_tags
-> 更新 entry_attachments
-> 重建該筆 entries_fts
```

搜尋結果以 `entry_id` 回查 `entries_index` 顯示。

## 8.3 migration 規則

建立 `IndexDatabaseMigrations`：

- 每次 schema 變更都要有 version
- migration 失敗時可 fallback 到 full rebuild

也就是：

```text
index database 是可丟棄、可重建的
```

## 8.4 建議索引

```sql
CREATE INDEX idx_entries_date ON entries_index(date);
CREATE INDEX idx_entries_updated_at ON entries_index(updated_at DESC);
CREATE INDEX idx_entries_deleted_date ON entries_index(is_deleted, date DESC);
CREATE INDEX idx_entries_favorite_updated ON entries_index(is_favorite, updated_at DESC);
CREATE INDEX idx_tags_tag_normalized ON entry_tags(tag_normalized);
CREATE INDEX idx_attachments_entry_id ON entry_attachments(entry_id);
CREATE INDEX idx_backup_history_created_at ON backup_history(created_at DESC);
```

## 8.5 查詢層邊界

Application layer 可依賴的查詢結果建議抽成獨立 model：

```text
TimelineListItem
CalendarDaySummary
SearchResultItem
BackupHistoryItem
AttachmentListItem
```

不要直接把 raw SQL row map 往 UI 丟。

## 9. App Lock 與金鑰管理流程

## 9.1 App 啟動狀態機

建議定義清楚狀態：

```text
uninitialized
locked
unlocking
unlocked
recovery_required
fatal_error
```

這會讓：

- 啟動畫面
- 生物辨識中斷
- Keystore 失效
- 備份前二次驗證

都比較好處理。

## 9.2 解鎖流程

日常解鎖：

```text
1. 驗證 PIN 或生物辨識
2. 由 Keystore 取回 device wrapping capability
3. 開啟 session
4. session 期間允許讀寫日記
```

session 應有：

- 閒置逾時
- 進背景鎖定
- 手動立即鎖定

## 9.3 Keystore 失效處理

偵測到舊 device slot 失效時，不要直接顯示致命錯誤。

應改為：

```text
進入 recovery_required 狀態
提示使用者輸入 Recovery Key
驗證成功後重建新的 device slot
```

這是核心 UX。

## 10. Recovery 流程實作規格

## 10.1 Onboarding 建立 Recovery Key

Use case：`SetupRecoveryKeyUseCase`

輸入：

- 已通過 App Lock 的 session

輸出：

- 新的 `recovery.json`
- 每個既有檔案新增 recovery slot

流程：

```text
1. 產生高熵隨機值
2. 編碼成人可抄寫格式
3. 顯示一次
4. 要求重新輸入驗證
5. 寫入 recovery.json
6. 對既有檔案補上 recovery slot
7. 完成後標記 recovery_enabled = true
```

## 10.2 單篇解鎖

Use case：`DecryptSingleEntryUseCase`

輸入：

- `.md.enc` 檔
- Recovery Key

輸出：

- 預覽模型
- 可選匯入或匯出結果

錯誤分類：

- 檔案不存在
- magic header 錯誤
- header JSON 損毀
- 無 recovery slot
- Recovery Key 錯誤
- ciphertext 驗證失敗

## 10.3 匯入加密 vault

Use case：`ImportEncryptedVaultUseCase`

流程：

```text
1. 驗證來源結構
2. 掃描 recovery metadata
3. 驗證 Recovery Key
4. 建立目標 vault
5. 複製或重寫加密檔
6. 建立新的 device slot
7. full rebuild index
8. 驗證筆數與附件數
```

### 10.4 還原覆蓋規則

需固定一條規則，避免 UI 混亂。

MVP 建議：

```text
若目前裝置已有 vault，還原前必須明確選擇：
1. 覆蓋現有 vault
2. 另建新 vault
```

不要隱性 merge。

## 11. Google Drive 備份格式與流程

## 11.1 `.jbackup` 內容

建議固定：

```text
backup_manifest.json
vault/recovery.json
vault/manifest.json.enc
vault/entries/**/*
vault/assets/**/*
vault/index/journal_index.sqlite
checksums.txt
```

`checksums.txt` 記錄每個檔案 SHA-256，可做還原驗證。

## 11.2 備份流程

Use case：`CreateBackupSnapshotUseCase`

流程：

```text
1. 鎖定寫入 session
2. flush autosave
3. 產生 backup staging 目錄
4. 複製 vault 檔案
5. 生成 backup_manifest.json
6. 生成 checksums.txt
7. 壓成 .jbackup
8. 上傳 Google Drive
9. 記錄最後成功備份時間
10. 解除鎖定
```

## 11.3 backup manifest 建議欄位

```json
{
  "backup_id": "bkp_...",
  "created_at": "2026-05-13T21:30:00+08:00",
  "app_version": "1.0.0",
  "vault_id": "vlt_...",
  "entry_count": 128,
  "asset_count": 342,
  "contains_index_snapshot": true,
  "format_version": 1
}
```

## 11.4 Google Drive 檔案保留策略

MVP 建議：

- 保留最近 10 份備份
- 使用者可手動刪除
- 上傳成功後才刪舊檔

## 16. 錯誤處理與訊息設計

所有錯誤不要只顯示技術例外字串，應分類成可理解訊息。

建議錯誤碼：

```text
E-LOCK-001 PIN 錯誤
E-LOCK-002 生物辨識取消
E-CRYPTO-001 header 格式錯誤
E-CRYPTO-002 Recovery Key 錯誤
E-CRYPTO-003 檔案完整性驗證失敗
E-BACKUP-001 Google 登入失敗
E-BACKUP-002 備份封包建立失敗
E-BACKUP-003 上傳失敗
E-INDEX-001 index 初始化失敗
E-INDEX-002 index 重建失敗
```

每個錯誤都應包含：

- 使用者可讀訊息
- 建議下一步
- 技術 log code

但 log 不得包含：

- 日記正文
- Recovery Key
- 明文標題全文

## 17. 測試補強清單

除了原計劃書已有內容，還建議補這些：

### 17.1 格式相容性測試

- 舊版 `schema_version=1` 是否仍可讀
- header 欄位缺漏是否正確報錯
- 多個 recovery slot 是否能正確選用

### 17.2 原子寫入測試

- 寫入中斷電後是否仍保有舊檔
- rename 前 crash 是否不污染正式檔

### 17.3 Index 重建測試

- 刪除 SQLite 檔後能否重建
- FTS 壞掉能否重建

### 17.4 備份一致性測試

- 備份時同時編輯是否被正確鎖住
- `.jbackup` 解壓後 checksum 是否一致

### 17.5 效能測試

- 1000 篇日記載入首頁時間
- 5000 張附件下備份耗時
- full rebuild index 耗時

## 18. MVP 驗收標準

MVP 完成不應只看功能存在，應看是否形成閉環。

驗收條件：

1. 新使用者第一次開啟 App，可在 1 分鐘內完成 PIN 與 Recovery Key 設定。
2. 使用者可建立、編輯、閱讀、刪除日記，且資料實際以 `.md.enc` 儲存。
3. 圖片可插入、儲存、顯示，且附件不是明文存放。
4. App 重啟後可正常由 SQLite 顯示時間線與日曆。
5. 刪除 index 後可由 vault 重建。
6. 單獨拿出一篇 `.md.enc`，可透過 Recovery Key 解鎖預覽。
7. 可建立 `.jbackup` 並上傳 Google Drive。
8. 新裝置可從 `.jbackup` + Recovery Key 還原。
9. Google Drive 與本機檔案中都不會出現明文日記內容。

## 19. 建議下一步

如果接下來要正式進專案實作，最合理的順序不是先做 UI，而是先做以下五件事：

4. 做 `index rebuild` prototype，確認資料可回推。
5. 再開始做 Flutter UI 與頁面流程。

這五件事如果先做對，後面 UI、備份、還原、Recovery 都會穩很多。

## 20. 本次整理結論

這次調整後，整體規格已經從「功能構想」收斂成比較穩定的工程方向，重點有四個：

1. 第一版就把資料契約定穩，不用之後再為了 vaultKey、索引角色、備份語意重構。
2. SQLite 不再只是輕量快取，而是完整查詢層，但仍維持可重建。
3. `.jbackup` 被明確定義為系統備份快照，避免與匯出格式混淆。
4. 優先順序改成 P0 / P1 / P2，代表功能都會做，但必須先完成安全與資料閉環。
