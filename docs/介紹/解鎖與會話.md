# 解鎖與會話

這份文件整理 Quill Diary 的解鎖與 session 行為，包括解鎖模式、可信裝置流程、背景逾時、resume 行為與設定頁切換。

它只講 session 與解鎖體驗，不展開加密檔案格式本身。

## 這份機制在做什麼

Quill Diary 不是只做一次解鎖就結束，而是維持一段「目前可讀寫日記庫」的 session。

這份機制主要解決幾件事：

- App 什麼時候算已解鎖
- 使用者回到前景時是否要重新驗證
- 可信裝置如何提供較順的重新進入體驗
- 什麼情況下必須退回復原金鑰路徑

## 三種解鎖模式

設定頁可在三種解鎖模式之間切換。

| 模式 | storage | 回前景驗證方式 | Keystore 槽 |
|------|---------|----------------|-------------|
| 無 | `none` | 不額外驗證，直接可信裝置 unwrap | `plain` |
| 裝置螢幕鎖 | `deviceLock` | 系統螢幕鎖對話框 | `deviceCredential` |
| 生物驗證 | `biometric` | 系統指紋或臉部驗證 | `biometric` |

補充：

- `none` 適合尚未設定螢幕鎖的裝置，但安全性較低；逾時後會直接以可信裝置 plain unwrap 恢復 session
- `deviceLock` 與 `biometric` 都要求裝置本身已設定螢幕鎖
- 切換至 `biometric` 時另須已登錄 strong biometric；日常解鎖時若事後移除生物辨識，仍可以螢幕鎖 fallback
- 即使選了生物驗證，系統驗證流程內仍可能允許使用螢幕鎖作為替代方式
- 解鎖模式偏好存放在 vault 外的 `app_lock_store.json`，與 vault 內索引欄位需透過 Keystore 同步保持一致

## Session 是什麼

session 代表目前 App 是否持有可用的解鎖狀態，能夠存取日記庫、草稿與搜尋索引。

高層來看：

- 有效 session 存在時，可讀寫正式日記、草稿與索引
- 鎖定後狀態改為 `locked`，關閉索引與 vault 資源存取；`activeVaultSessionProvider` 不再暴露 session
- 鎖定狀態下不能直接存取這些敏感資料
- 首次安裝、尚未建立復原金鑰時，狀態可能是 `unlocked` 但 `session == null`；這是引導建立復原金鑰的 onboarding，尚不能讀寫日記庫

## App 啟動與狀態

目前僅支援 Android；非 Android 平台 bootstrap 會進入 `fatalError`。

高層狀態由 `AppLockStatus` 描述，常見值包括：

| 狀態 | 意義 |
|------|------|
| `uninitialized` | 啟動中，尚未完成 bootstrap |
| `unlocking` | 正在可信裝置或復原金鑰解鎖 |
| `unlocked` | 已解鎖；是否可讀寫取決於 `session` 是否為 null |
| `locked` | 已鎖定，需重新驗證或手動解鎖 |
| `recoveryRequired` | 必須輸入復原金鑰 |
| `fatalError` | 無法繼續啟動的致命錯誤 |

Provider 鏈概略為：

- `appSessionProvider` 持有執行期狀態
- 若仍為 `uninitialized`，`appStartupProvider` 會呼叫 `bootstrapAppSession()`
- UI 主要 watch `effectiveAppSessionProvider`（合併啟動與執行期視圖）
- `activeVaultSessionProvider` 只在 `unlocked` 且 `session != null` 時暴露有效 session

冷啟動時，若本機已有可信裝置，`bootstrapAppSession()` 會直接呼叫 `unlock()`；這與之後 lifecycle resume 的自動解鎖是不同路徑。

## 冷啟動與 lifecycle

bootstrap 結束後，程式會進入 `SessionStartupPhase` 週期，用來隔離冷啟動與後續 lifecycle resume：

| 階段 | 意義 |
|------|------|
| `bootstrapping` | bootstrap 進行中；lifecycle 背景事件不算數 |
| `awaitingFirstBackground` | bootstrap 已結束，但尚未進入一般 lifecycle 流程 |
| `resumeUnlockArmed` | 前景穩定後才允許記錄真實背景，並排程 resumed 自動解鎖 |

高層流程：

1. `bootstrapAppSession()` 完成後進入 `awaitingFirstBackground`
2. bootstrap 成功且 `unlocked`、有有效 session 時，App 前景穩定約 **500ms** 後才 `armLifecycleResumeUnlock()`
3. 進入 `resumeUnlockArmed` 後，第一次真實的 `paused` / `hidden` 才會被記成後續 resumed 的背景依據
4. 冷啟動後第一次 `resumed` **不會**只因 lifecycle 而自動解鎖；須先完成上述 arm，且之後有過有效背景切換

補充：

- 還原流程應直接呼叫 `bootstrapAppSession()` 或 `finishRestoreSession()`，避免與 `appStartupProvider` 並行解鎖

## 可信裝置與加密的關係

可信裝置解鎖不是逐檔透過 Keystore 解開 `fileKey`，而是先在 **vault 層** 取得 `recoveryWrapKey`。

`hasTrustedKey` 須同時滿足 Android Keystore 槽與 `flutter_secure_storage` 內的 `WrappedRecoveryKeyRecord`；缺一即視為無可信裝置。

高層流程：

1. `VaultRepository.openTrustedSessionEnsuringKeystore` 透過 `DeviceKeyManager` 與 Android Keystore unwrap vault 層 wrapped recovery key（必要時同步 Keystore 遷移）
2. session 取得 `recoveryWrapKey` 與 `deviceSlotId`
3. `ensureIndexReady` 開啟搜尋索引並同步標籤樣式（日常解鎖不會全量重建索引，見 [索引資料庫.md](./索引資料庫.md)）
4. 之後讀寫 LDJ2 檔案、草稿與索引時，都用這把 key 解開各檔 header 的 recovery slot

加密檔案結構與兩層金鑰細節，請看 [加密格式.md](./加密格式.md)。

## 核心元件分工

| 元件 | 職責 |
|------|------|
| `AppSessionController` | 管理 session 狀態、`lockReason`，並呼叫 `VaultRepository.openTrustedSessionEnsuringKeystore` |
| `SessionInactivityWatchdog` | 追蹤 `paused` / `hidden` 背景時間；`resumed` 後約 500ms 前景穩定期，用於清除背景計時或判定逾時 |
| `AppLifecycleSessionBridge` | 接 App 生命週期與使用者觸碰；鎖定且可 resume 解鎖時，約 350ms 後觸發 `unlock(source: lifecycleResume)` |
| `AppLockService` | 保存解鎖模式至 `app_lock_store.json`，並以 `getDeviceAuthCapabilities()` 檢查裝置能力 |
| `unlock_mode_policy.dart` | 解鎖模式能力檢查（migrate / unlock）、可信狀態一致性比對、鎖定提示文案 |
| `UnlockModeChangeService` | 安全切換解鎖模式（Keystore 同步成功後才寫入偏好） |
| `DeviceKeyManager` | 管理 Android Keystore 的 wrap / unwrap，並清除 inactive 金鑰 |
| `VaultRepository` | 建立可信裝置 session，並確認 Keystore 狀態是否符合目前解鎖模式 |

這幾個元件一起決定目前是否能以可信裝置路徑重新進入日記庫。

## Resume 行為

App 回到前景時，不會只因為收到一次 `resumed` 就立刻信任目前狀態。現在流程分成兩段：

**背景計時（`SessionInactivityWatchdog`）**

- `resumed` 後先等待約 **500ms** 前景穩定期
- 若這段期間 App 仍持續留在前景，就清掉背景計時並保留目前 session
- 若 `resumed` 時已超過背景逾時，則先進入 `locked + lockReason: inactivity`，並關閉索引等 vault 資源

**自動解鎖（`AppLifecycleSessionBridge`）**

- 須已進入 `resumeUnlockArmed`，且本次啟動週期內曾有效進背景
- 鎖定且 `shouldUnlockOnResume`（`lockReason` 不是 `manual`）時，約 **350ms** 後呼叫 `unlock(source: lifecycleResume)`
- `none` 模式：靜默可信裝置 unwrap
- `deviceLock` / `biometric` 模式：跳出系統驗證對話框

如果 resumed 自動觸發的驗證沒有成功：

- 使用者取消或驗證逾時：App 維持 `locked`，並**保留原本的 `lockReason`**（例如仍為 `inactivity`）；這次 `resumed` 不立刻重試
- 真正驗證失敗：App 會進入 `locked`，`lockReason: authFailed`
- 金鑰失效或格式不符等永久失敗：可能清除可信裝置並進入 `recoveryRequired`
- 使用者可隨時按「重新驗證」再次觸發
- `lockReason: manual`（例如使用者手動鎖定，或在設定頁取消 `unlocking`）也不會在同一次 resumed 內自動重試
- 無螢幕鎖、嘗試次數過多等情況會回傳對應錯誤碼

補充：

- 設定頁在 `unlocking` 狀態可選擇「取消並改用手動驗證」
- 這會讓 session 回到 `locked + lockReason: manual`，之後由使用者自行按「重新驗證」
- 因背景逾時鎖定後，lifecycle 解鎖成功時可能還原鎖定前的路由

## 背景逾時

session 不會永久保持有效。當 App 進背景過久，就必須重新驗證。

- 預設門檻為 3 分鐘
- 可在個人化頁調整為 `1 / 3 / 5 / 10 分鐘`
- **僅** `paused` / `hidden` 開始計時；`inactive`（例如下拉通知欄）不計時
- `SessionInactivityWatchdog` 以 Timer 追蹤；`resumed` 後會先等待約 500ms 前景穩定期
- 若 App 在穩定期內仍持續留在前景，就清掉背景計時；不要求使用者一定先點一下畫面
- `onPointerDown`（僅在已解鎖時）仍會提早清掉背景狀態，作為額外保險
- 一旦逾時，狀態改為 `locked` + `lockReason: inactivity`，並關閉索引等 vault 資源；`activeVaultSessionProvider` 不再暴露 session
- 回到前景後，若符合 lifecycle 條件，`AppLifecycleSessionBridge` 會自動呼叫 `unlock()`；實際是否跳系統驗證由解鎖模式決定

例外：若背景期間仍有敏感操作進行中（`runSensitiveTask` 或 `runBackgroundSafeTask`，例如備份、還原、匯入匯出），`_activeSensitiveTasks > 0` 時**不會**觸發背景逾時鎖定，避免長時間作業中途被強制鎖定。鎖定時若仍有進行中敏感任務，關閉 vault 資源會延後執行。

這個行為的重點是：可信裝置只是在一段有效時間內降低重新進入成本，不是永久免驗證。

## 切換解鎖模式

設定頁可以在三種解鎖模式之間切換，但不是任意切。

切換流程重點：

1. 必須已建立復原金鑰
2. 必須處於已解鎖 session（鎖定時設定頁會停用切換）
3. 切換前先以 `migrate` 情境檢查裝置能力（螢幕鎖；生物驗證另需 strong biometric）
4. 使用者須通過系統驗證後，`ensureKeystoreMatchesUnlockMode` 才會 re-wrap
5. Keystore 同步成功後才寫入新偏好；成功 re-wrap 後會清除其他 auth kind 的 inactive 金鑰

這代表模式切換不只是 UI 選項切換，還包含底層 Keystore 狀態調整。

## 什麼時候會退回復原金鑰

可信裝置解鎖只是便利路徑，不是最終的資料所有權依據。

以下情況會讓使用者不能只靠可信裝置繼續使用，而必須退回復原金鑰路徑：

- 可信裝置狀態失效
- Keystore 狀態與目前解鎖模式不一致
- 還原後偵測到目前可信狀態與日記庫不再匹配
- 進入 `recoveryRequired` 狀態

這也是為什麼復原金鑰始終是必要前提，而不是可選附加功能。
