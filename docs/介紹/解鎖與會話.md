# 解鎖與會話

這份文件整理 Quill Diary 的解鎖與 session 行為，包括解鎖模式、可信裝置流程、背景逾時、resume 行為與設定頁切換。

它只講 session 與解鎖體驗，不展開加密檔案格式本身。

## 這份機制在做什麼

Quill Diary 不是只做一次解鎖就結束，而是維持一段「目前可讀寫日記庫」的 session。

這份機制主要解決幾件事：

- App 什麼時候算已解鎖
- 使用者回到前景時是否要重新驗證
- 可信裝置如何提供較順的重新進入體驗
- 什麼情況下必須退回復原金鑰路徑

## 三種解鎖模式

設定頁可在三種解鎖模式之間切換。

| 模式 | storage | 回前景驗證方式 | Keystore 槽 |
|------|---------|----------------|-------------|
| 無 | `none` | 不額外驗證，直接可信裝置 unwrap | `plain` |
| 裝置螢幕鎖 | `deviceLock` | 系統螢幕鎖對話框 | `deviceCredential` |
| 生物驗證 | `biometric` | 系統指紋或臉部驗證 | `biometric` |

補充：

- `none` 適合尚未設定螢幕鎖的裝置，但安全性較低；逾時後會直接以可信裝置 plain unwrap 恢復 session
- `deviceLock` 與 `biometric` 都要求裝置本身已設定螢幕鎖
- 切換至 `biometric` 時另須已登錄 strong biometric；日常解鎖時若事後移除生物辨識，仍可以螢幕鎖 fallback
- 即使選了生物驗證，系統驗證流程內仍可能允許使用螢幕鎖作為替代方式

## Session 是什麼

session 代表目前 App 是否持有可用的解鎖狀態，能夠存取日記庫、草稿與搜尋索引。

高層來看：

- 有效 session 存在時，可讀寫正式日記、草稿與索引
- session 清掉後，App 會回到鎖定狀態
- 鎖定狀態下不能直接存取這些敏感資料

## 可信裝置與加密的關係

可信裝置解鎖不是逐檔透過 Keystore 解開 `fileKey`，而是先在 **vault 層** 取得 `recoveryWrapKey`。

高層流程：

1. `VaultRepository.openTrustedSession` 透過 `DeviceKeyManager` 與 Android Keystore unwrap vault 層 wrapped recovery key
2. session 取得 `recoveryWrapKey` 與 `deviceSlotId`
3. 之後讀寫 LDJ2 檔案、草稿與索引時，都用這把 key 解開各檔 header 的 recovery slot

加密檔案結構與兩層金鑰細節，請看 [加密格式.md](./加密格式.md)。

## 核心元件分工

| 元件 | 職責 |
|------|------|
| `AppSessionController` | 管理 session 狀態、`lockReason`，並呼叫 `VaultRepository.openTrustedSession` |
| `SessionInactivityWatchdog` | 追蹤 `paused` / `hidden` 背景時間，並在 `resumed` 後做短暫前景穩定期判定 |
| `SessionLifecycleBinding` | 接 App 生命週期與使用者觸碰，並在前景穩定後觸發自動 reauth |
| `AppLockService` | 保存解鎖模式，並以 `getDeviceAuthCapabilities()` 檢查裝置能力 |
| `unlock_mode_policy.dart` | 解鎖模式能力檢查（migrate / unlock）、可信狀態一致性比對、鎖定提示文案 |
| `UnlockModeChangeService` | 安全切換解鎖模式（Keystore 同步成功後才寫入偏好） |
| `DeviceKeyManager` | 管理 Android Keystore 的 wrap / unwrap，並清除 inactive 金鑰 |
| `VaultRepository` | 建立可信裝置 session，並確認 Keystore 狀態是否符合目前解鎖模式 |

這幾個元件一起決定目前是否能以可信裝置路徑重新進入日記庫。

## Resume 行為

App 回到前景時，不會只因為收到一次 `resumed` 就立刻信任目前狀態。現在流程是：

- `SessionInactivityWatchdog` 先在 `resumed` 後等待一個很短的前景穩定期（約 500ms）
- 若這段期間 App 仍持續留在前景，就清掉背景計時並保留目前 session
- 若 `resumed` 時已超過背景逾時，則先進入 `locked + lockReason: inactivity`

若因背景逾時而鎖定，`SessionLifecycleBinding` 會在下一幀呼叫 `unlock()`：

- `none` 模式：靜默可信裝置 unwrap
- `deviceLock` / `biometric` 模式：跳出系統驗證對話框

如果使用者取消或驗證失敗：

- App 維持 `locked`，`lockReason: authFailed`
- 使用者可隨時按「重新驗證」再次觸發
- `lockReason: manual`（例如使用者手動鎖定，或在設定頁取消 `unlocking`）也不會自動重試
- 無螢幕鎖、驗證逾時、嘗試次數過多等情況會回傳對應錯誤碼，不再一律視為「取消」

補充：

- 設定頁在 `unlocking` 狀態可選擇「取消並改用手動驗證」
- 這會讓 session 回到 `locked + lockReason: manual`，之後由使用者自行按「重新驗證」

## 背景逾時

session 不會永久保持有效。當 App 進背景過久，就必須重新驗證。

- 預設門檻為 3 分鐘
- 可在個人化頁調整為 `1 / 3 / 5 / 10 分鐘`
- **僅** `paused` / `hidden` 開始計時；`inactive`（例如下拉通知欄）不計時
- `SessionInactivityWatchdog` 以 Timer 追蹤；`resumed` 後會先等待短暫前景穩定期
- 若 App 在穩定期內仍持續留在前景，就清掉背景計時；不要求使用者一定先點一下畫面
- `onPointerDown` 仍會提早清掉背景狀態，作為額外保險
- 一旦逾時，既有 session 會被清除，狀態為 `locked` + `lockReason: inactivity`
- 回到前景後自動呼叫 `unlock()`；實際是否跳系統驗證由解鎖模式決定

例外：若背景期間仍有敏感操作進行中（`runSensitiveTask`，例如備份、還原、匯入匯出），`_activeSensitiveTasks > 0` 時**不會**觸發背景逾時鎖定，避免長時間作業中途被強制鎖定。

這個行為的重點是：可信裝置只是在一段有效時間內降低重新進入成本，不是永久免驗證。

## 切換解鎖模式

設定頁可以在三種解鎖模式之間切換，但不是任意切。

切換流程重點：

1. 必須已建立復原金鑰
2. 必須處於已解鎖 session（鎖定時設定頁會停用切換）
3. 切換前先以 `migrate` 情境檢查裝置能力（螢幕鎖；生物驗證另需 strong biometric）
4. 使用者須通過系統驗證後，`ensureKeystoreMatchesUnlockMode` 才會 re-wrap
5. Keystore 同步成功後才寫入新偏好；成功 re-wrap 後會清除其他 auth kind 的 inactive 金鑰

這代表模式切換不只是 UI 選項切換，還包含底層 Keystore 狀態調整。

## 什麼時候會退回復原金鑰

可信裝置解鎖只是便利路徑，不是最終的資料所有權依據。

以下情況會讓使用者不能只靠可信裝置繼續使用，而必須退回復原金鑰路徑：

- 可信裝置狀態失效
- Keystore 狀態與目前解鎖模式不一致
- 還原後偵測到目前可信狀態與日記庫不再匹配
- 進入 `recoveryRequired` 狀態

這也是為什麼復原金鑰始終是必要前提，而不是可選附加功能。
