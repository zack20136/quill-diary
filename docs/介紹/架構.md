# 架構

這份文件整理 Quill Diary 的靜態設計，包括產品目標、程式分層、主要模組與本機資料落點。

它只回答兩件事：

- 東西放在哪裡
- 哪個模組負責什麼

執行流程、解鎖細節、索引生命週期與備份操作，請看各自主題文件。

## 產品目標

Quill Diary 的架構核心很明確：把私人日記當成一個本機加密資料系統來設計，而不是單純的文字編輯器。

目前整體設計圍繞這幾個目標：

- 日記與附件以加密格式存放在本機日記庫
- 使用者可透過可信裝置或復原金鑰重新取得存取權
- 搜尋索引只在解鎖 session 期間開啟
- 搜尋要能涵蓋標題、標籤與正文（索引儲存的是 Markdown 正規化後的可搜尋文字）
- 完整備份與可攜式匯出分開設計，避免用途混淆

補充：目前僅支援 Android。

## 專案分層

| 目錄 | 角色 |
|------|------|
| `lib/app/` | App shell、router、theme 與整體進入點 |
| `lib/presentation/` | 頁面、widget、presentation state 與 feature-local provider |
| `lib/application/` | flow、controller、coordinator 與跨模組流程協調 |
| `lib/domain/` | 穩定模型、value object 與跨模組共用的核心型別 |
| `lib/infrastructure/` | 加密、儲存、索引、裝置安全與外部整合 |
| `lib/shared/` | 跨功能共用的 UI primitive、平台支援與純工具函式 |
| `lib/l10n/` | 在地化字串與 ARB 文案來源 |

這個分層的原則是：

- `presentation` 負責使用者看得到的頁面與互動
- `application` 負責把 session、restore、settings、editor 等流程協調成可執行行為
- `domain` 負責穩定資料模型
- `infrastructure` 負責把資料真正存下來、加密、索引或搬移
- `shared` 只放跨功能共用但不屬於單一 bounded context 的內容

## 主要功能模組

| 模組 | 負責內容 |
|------|------|
| Session | 啟動、鎖定、解鎖、timeout、resume 與解鎖模式切換 |
| Home | 日記列表、日曆、標籤、總覽與搜尋；總覽另提供範圍回顧 HTML 匯出 |
| Editor | 建立與編輯日記、附件操作、本地草稿與正式儲存 |
| Settings | 復原金鑰、備份還原、個人化、介紹、支持與法律隱私相關入口 |

這些模組不直接各自管理底層儲存格式，而是透過 `infrastructure` 提供的元件協作。

補充：

- `Settings` 頁 AppBar 右側提供三個子頁入口：`個人化`、`介紹`、`支持`
- `個人化` 處理語言、主題、自動鎖定、圖片品質與日記排版
- `介紹` 提供產品、解鎖、加密、索引、編輯器與備份等說明頁
- `支持` 為 Google Play Billing 的一次性支持頁，不解鎖任何額外功能

## `lib/infrastructure/storage/` 的角色

`lib/infrastructure/storage/` 是這個專案最核心的資料層之一，負責日記庫、草稿、備份、匯入匯出與外部交付。

| 區塊 | 職責 |
|------|------|
| 根目錄 | 日記庫讀寫、封存、資料轉移、外部資料夾設定與路徑策略 |
| `shared/` | zip 解封、外部資料夾挑選與 Android SAF 檔案交付 |
| `portable/` | 完整備份 zip、Markdown / HTML 可攜式匯入匯出 |
| `import/<source>/` | 其他來源備份的匯入邏輯，目前包含 `easy_diary/` |

如果只是想快速對照類別與檔案，請看 [模組參考.md](./模組參考.md)。

## 核心模組邊界

### `VaultRepository`

- 管理日記庫初始化與 `recovery.json` metadata
- 負責加密日記與附件的讀寫、可信裝置 session 與復原金鑰解鎖
- 協調正式儲存後的搜尋索引同步
- 還原後 session 啟動細節見 [解鎖與會話.md](./解鎖與會話.md)

### `VaultTransferService`

- 管理完整備份、還原、可攜式匯入匯出
- 處理外部資料夾交付流程
- 負責資料搬移與封裝，不負責日常編輯流程

### `IndexDatabaseManager`

- 管理搜尋索引資料庫的開啟、關閉與重建
- 負責索引層，不直接擁有日記主資料

### `EditorDraftStore`

- 管理 `drafts/` 目錄中的本地加密草稿
- 管理編輯期間的待上傳附件暫存
- 只處理草稿，不直接等同正式日記庫

## 本機資料落點

| 路徑 | 內容 |
|------|------|
| `vault/recovery.json` | 明文復原金鑰 metadata 與 KDF 描述 |
| `vault/manifest.json.enc` | 加密 manifest |
| `vault/entries/**/*.md.enc` | 加密日記內容 |
| `vault/assets/**/*.enc` | 加密附件 |
| `vault/tag_styles.json` | 明文標籤樣式與目錄資料（非 LDJ2） |
| `vault/pinned_entries.json` | 首頁釘選狀態；屬於正式日記庫資料，會隨完整備份與還原保留 |
| `index/journal_index.sqlite` | 解鎖期間才開啟的搜尋索引 |
| `drafts/{draftKey}/` | 本地加密草稿（`draft.json.enc`）與 LDJ2 加密的待上傳附件暫存（`pending/*.enc`） |
| `backups/` | 本機完整備份 `backup_*.zip`（最多保留 5 份） |
| `app_preferences.json` | 存放於 vault 外的個人化偏好（主題、語言、自動鎖定、圖片品質、排版） |
| `app_lock_store.json` | 存放於 vault 外的解鎖模式偏好 |

其中：

- 以上相對路徑都在 `{appSupport}/quill_diary/` 下（與 [索引資料庫.md](./索引資料庫.md) 的路徑前綴一致）
- `vault/` 是正式資料本體
- `index/` 是衍生搜尋資料
- `drafts/` 是編輯中的暫存區，不屬於正式日記庫

## 格式不符時的高層處理

這個專案的原則不是硬撐錯誤格式，而是回到可重建的穩定狀態。

- 可信裝置狀態失效時：清除本機可信存取，要求使用復原金鑰重新建立
- 搜尋索引格式或 schema 不符時：刪除索引並重新建構

這也反映了架構上的分工：正式資料、裝置信任狀態與搜尋索引是不同層次的資料，各自可重建或重設。
