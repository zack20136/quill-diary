# 日記編輯器

這份文件整理 Quill Diary 目前的日記編輯器行為，重點放在編輯模式、草稿、附件、任務清單、預覽、session 邊界與正式儲存。

它描述的是目前程式實際怎麼運作，不是理想設計，也不展開 Markdown 語法教學。

## 先講結論

- 正式資料只存在 `vault/`；`drafts/` 只是尚未提交的本地加密暫存。
- 既有日記預設先進檢視模式，新建日記則直接進編輯模式。
- 編輯中的欄位變更會自動寫入本地草稿，但草稿不會進搜尋索引，也不會進完整備份。
- 內文含任務清單時，編輯器會切到混合區塊模式；預覽模式中的任務勾選會直接回寫並正式儲存該篇日記。
- session 鎖定後不能再讀寫草稿或正式資料；暫時解密出的附件預覽檔也會被清掉。

## 能做什麼

| 功能 | 目前行為 |
|------|------|
| 新建日記 | 直接進編輯模式，先以本地草稿承接未完成內容 |
| 編輯既有日記 | 預設先看檢視模式，按編輯才進入編輯模式 |
| 自動草稿保存 | 編輯中的標題、日期、時間、標籤、內文、附件變更都會排程寫入草稿 |
| 草稿還原 | 重開同一篇日記或未完成新建時，若有草稿會先跳出詢問 |
| Markdown 內文 | 純文字內容用一般文字編輯；含任務清單時切到混合區塊編輯 |
| 任務清單 | 可插入 `- [ ]` / `- [x]`，每個任務可個別編輯、勾選與拖曳排序 |
| 預覽模式 | 可切到 Markdown 預覽；空內容會顯示「尚未輸入內容」 |
| 預覽勾選 | 預覽中的任務勾選不是只改 UI，而是會立即回寫 markdown 並正式儲存 |
| 標籤 | 可直接輸入 CSV，也可開標籤編輯對話框整理 |
| 圖片附件 | 可從系統圖片選取器加入多張圖片 |
| 一般附件 | 可透過檔案選取器附加任意檔案 |
| 附件排序 | 圖片附件可在編輯器內重新排序；已儲存圖片與待儲存圖片會一起排序 |
| 附件移除 | 可移除既有附件，也可移除尚未正式寫入的 pending 附件 |
| 刪除日記 | 既有日記在檢視或編輯模式都可刪除 |
| 未儲存標記 | 既有日記只要存在本地草稿，列表與檢視模式都會顯示「未儲存」 |

## 進入與離開編輯器

| 情境 | 目前行為 |
|------|------|
| 開啟新日記 | 直接進編輯模式 |
| 開啟既有日記 | 先顯示檢視模式 |
| 按編輯 | 切到編輯模式，並以目前 entry 建立 dirty 比對基準 |
| 按關閉（檢視模式） | 直接離開頁面 |
| 按關閉（編輯模式，無變更） | 清掉本地草稿；新建頁直接離開，既有日記回檢視模式 |
| 按關閉（編輯模式，有變更） | 跳出確認是否放棄變更；確認後刪除草稿並離開或回檢視模式 |
| 儲存成功 | 寫入 `vault/`、刪掉草稿、刷新首頁與索引相關 provider；新建日記會改用正式 `entryId` 重新導向 |

草稿 key 規則：

- 新建日記固定用 `__new__`
- 既有日記直接用 `entryId`

這表示 App 同時間只會保留一份「未正式儲存的新日記草稿」。

## 欄位與保存規則

### 標題、日期、時間

- 標題可以留空；真正的保存條件是「標題或內文至少要有一個非空」。
- 日期欄位保存的是 `DateOnly` 字串值，格式為 `YYYY-MM-DD`。
- 時間使用 24 小時制。
- 既有日記若改日期或時間，會重組 `createdAt` 的年月日與時分，但保留原本的秒、毫秒與微秒。
- 新建日記第一次正式儲存時，秒以下時間取當下 `DateTime.now()`。

### 標籤

- 標籤欄位以逗號分隔。
- 每個標籤會先 `trim`，連續空白會折疊成單一空白。
- 會依正規化後的文字去重複，但保留第一次輸入的顯示形式。
- 標籤順序會納入 dirty 比對，所以換順序也算修改。

### 內文

- 內文編輯前後都以 Markdown 字串為單一資料來源。
- 若 Markdown 內含 checkbox line，編輯器會切到混合區塊模式。
- 純文字模式下儲存前會移除前後空白。
- 任務清單模式下儲存前會做正規化，並保留編輯所需的尾端空白文字行，例如：

```md
前言
- [ ] 任務三
```

會保存成：

```md
前言
- [ ] 任務三
```

加上一個結尾換行。

- 刪掉最後一個任務項目後，編輯器會回到純文字模式。

## 任務清單與預覽

### 混合區塊模式

- checkbox line 的判斷規則是 `^\s*-\s*\[([ xX])\]`.
- 可在目前游標位置插入新任務項目。
- 若在文字中間插入任務項目，後續文字會保留在新任務之後，不會多插一個空白段落。
- 任務與文字行都可重組成同一份 markdown，不另外維護第二份資料模型。
- 編輯器會盡量保留 line id，避免因輕微重繪讓整列 widget 被當成新項目。

### 預覽模式

- 預覽模式使用 [`editor_markdown_preview.dart`](../../../lib/presentation/editor/widgets/editor_markdown_preview.dart) 顯示 Markdown。
- 只有內文含任務清單時，預覽中的 checkbox 才是可互動的。
- 點選預覽中的任務勾選後：
  1. 先樂觀更新 `_bodyController`
  2. 呼叫 `EditorFlowController.saveEntry(...)`
  3. 立即把該變更正式寫回 `vault/`
- 這條路徑不是草稿保存，所以它會同步更新正式 entry 與索引。

## 草稿如何運作

草稿是編輯流程的暫存層，不是正式日記庫的延伸。

### 寫入時機

- 標題、日期、時間、標籤、內文、附件清單變更時，會排程寫草稿。
- App 進入 `hidden` 或 `paused` 生命週期時，會立即嘗試落草稿。
- 關閉編輯器前會先強制把排隊中的草稿寫完，再決定是否需要跳確認。
- 若目前是新建日記且內容完全為空，不會建立草稿。

### 寫入內容

`draft.json.enc` 內包含：

- 標題
- 日期字串
- 時分
- 正規化後的標籤清單
- 正規化後的 Markdown 內文
- 保留中的既有附件 id 清單
- pending 附件的相對路徑、MIME type、原始檔名
- provisional entry id
- 草稿建立與更新時間

### 儲存位置

| 路徑 | 內容 |
|------|------|
| `drafts/{draftKey}/draft.json.enc` | LDJ2 加密的草稿 JSON |
| `drafts/{draftKey}/pending/` | 尚未正式寫入日記庫的附件暫存，檔名為 `*.enc` |

路徑策略定義在 [`vault_path_strategy.dart`](../../../lib/infrastructure/storage/vault_path_strategy.dart)。

### 加密與預覽

- 草稿 JSON 與 `pending/` 附件都用 LDJ2 加密。
- 寫草稿時需要目前有效的 `UnlockedVaultSession`。
- `vaultId` 不一致或 session 缺少 `recoveryWrapKey` 時，草稿無法正常讀寫。
- UI 不直接讀 `.enc` 檔；預覽附件前會先把檔案短暫解密到系統暫存目錄。
- 這些暫時明文檔會快取在 `Directory.systemTemp/quill_diary_draft_preview/...` 下。

### 還原與清理

- 開頁後若偵測到草稿，會先詢問是否還原。
- 若使用者拒絕還原，該草稿會立刻刪除。
- 若是新建草稿但內容其實為空，也會直接刪除，不再詢問。
- 每次重寫草稿時，都會順便清掉 record 已不再引用的 pending 附件。
- 正式儲存成功後，對應草稿資料夾會被刪除。

## 附件流程

### 圖片

- 圖片選取走 `image_picker`。
- 會先嘗試 `pickMultiImage()`；若失敗才退回單張 `pickImage(ImageSource.gallery)`。
- 取回圖片後，會依個人化設定的 `ImageCompressPreset` 決定是否先壓縮。
- GIF 不會被轉 JPEG；其他被壓縮的圖片會轉成 JPEG 暫存後再寫入 pending。
- pending 寫入完成後，編輯器會再 materialize 一份暫時明文圖檔供預覽使用。

### 一般檔案

- 一般檔案選取走 `file_picker`。
- 檔案會直接寫進 `drafts/{draftKey}/pending/*.enc`。
- 預覽或後續交互使用的是 materialize 後的暫時明文路徑，不是加密原檔。

### 正式儲存

- 正式儲存時，`EditorFlowController.saveEntry(...)` 會把：
  - 保留中的既有附件 id
  - 新增的 pending 附件
  一起交給 `VaultRepository.saveEntry(...)`
- 草稿 pending 附件在這一步才真正進入 `vault/`，成為正式附件。

## Session 與敏感資料邊界

- editor 的讀寫前提是 session 已解鎖。
- session 進入 locked 後：
  - 若當下正在編輯，會先嘗試把內容落草稿
  - `EditorDraftStore.clearAllMaterializedPendingFiles()` 會清掉所有暫時解密附件
  - vault 已解鎖資源也會關閉
- 頁面重新取得有效 session 後，會再依草稿狀態決定是否提示還原。
- 若鎖定發生在編輯中，頁面會保留「曾在編輯」的上下文，避免解鎖回來時直接丟失可還原機會。

## 與其他模組的邊界

| 模組 | 邊界 |
|------|------|
| 搜尋索引 | 草稿不進索引；只有 `saveEntry` 的正式儲存會刷新相關 provider 與索引讀取 |
| 完整備份 | `backup_*.zip` 只封裝 `vault/` 正式資料，不包含 `drafts/` |
| 還原 | 還原備份後會清除所有本地草稿與 pending 附件，避免和還原後的 `vault/` 不一致 |
| 個人化設定 | 圖片壓縮 preset 與排版設定會直接影響編輯器行為 |
| 列表 UI | `editorDraftKeysProvider` 會掃描 `drafts/`，讓列表與檢視模式顯示「未儲存」標記 |
