# 解鎖與會話

這份文件整理 Quill Diary 目前的解鎖與 session 行為，包括冷啟動、trusted device、背景逾時、resume 自動解鎖、restore 後重啟，以及設定頁切換解鎖模式。

它只講「什麼時候算已解鎖、誰可以繼續讀 vault、什麼情況要退回 Recovery Key」，不展開 `LDJ2` 檔案格式本身。

## 先講結論

- `session` 是敏感資料存取的共同入口；沒有有效 `UnlockedVaultSession`，就不應該讀寫 vault、草稿或索引。
- `AppLockStatus.unlocked` 不保證一定可讀資料；還要同時有 `session != null`。
- trusted device 解的是 vault 層的 `recoveryWrapKey`，不是逐檔直接解 `fileKey`。
- 冷啟動 bootstrap、背景逾時鎖定、resume 自動解鎖、restore 後重啟，是同一套 session 狀態機的不同入口。

## 這套機制在做什麼

Quill Diary 不是只做一次解鎖就結束，而是維持一段「目前可讀寫日記庫」的執行期 session。

它主要在解決幾件事：

- App 啟動時要走 trusted device 還是 Recovery Key
- App 進背景多久後要上鎖
- 回到前景時可不可以自動重新驗證
- 還原後可不可以沿用原本 session
- 使用者切換解鎖模式時，Keystore 保護是否也同步完成

## 解鎖模式

設定頁可在三種解鎖模式之間切換。

| 模式 | 儲存值 | 回前景驗證方式 | Keystore auth kind |
|------|--------|----------------|--------------------|
| 無 | `none` | 不跳系統驗證，直接 trusted device unwrap | `plain` |
| 裝置螢幕鎖 | `deviceLock` | 系統螢幕鎖驗證 | `deviceCredential` |
| 生物驗證 | `biometric` | 系統生物驗證 | `biometric` |

補充：

- `none` 不是「不用 trusted device」；它只是把 trusted device 包裝改成 `plain` 模式。
- `deviceLock` 與 `biometric` 都要求裝置具備 device credential。
- 切到 `biometric` 時，`migrate` 檢查要求 strong biometric；但日常 `unlock` 檢查只要求 device credential 可用，因此系統仍可能允許螢幕鎖 fallback。
- 解鎖模式偏好獨立存放在 vault 外的 `app_lock_store.json`。
- vault 內索引資料庫還會同步記錄目前 Keystore 保護後綴；兩邊不一致時，會觸發 Keystore migration / rewrap。

## Session 是什麼

目前 session 狀態由 `AppSessionState` 表示，核心欄位是：

- `status`
- `session`
- `message`
- `lockReason`

高層語意如下：

- `status == unlocked` 且 `session != null`：可讀寫 vault、草稿與索引。
- `status == unlocked` 但 `session == null`：通常是尚未建立 Recovery Key 的 onboarding 狀態，還不能讀正式資料。
- `status == locked`：必須重新驗證；`activeVaultSessionProvider` 不再提供 session。
- `status == recoveryRequired`：trusted device 路徑不可用，必須改走 Recovery Key。

目前常見 `AppLockStatus`：

| 狀態 | 意義 |
|------|------|
| `uninitialized` | 啟動中，尚未完成 bootstrap |
| `unlocking` | 正在 trusted device 或 Recovery Key 解鎖 |
| `unlocked` | 已進入解鎖狀態；是否真的能讀資料取決於 `session` |
| `locked` | 已鎖定，等待重新驗證 |
| `recoveryRequired` | 必須輸入 Recovery Key |
| `fatalError` | 啟動或驗證流程無法繼續 |

Provider 邊界：

- `appSessionProvider`：執行期主狀態。
- `sessionStartupProvider`：只在 `uninitialized` 時執行 bootstrap。
- `effectiveAppSessionProvider`：合併啟動中與執行期視圖，UI 主要 watch 它。
- `activeVaultSessionProvider`：只在 `unlocked && session != null` 時輸出 session。
- `indexQueryableVaultSessionProvider`：正常情況跟著有效 session 走，但在 `unlocking` 過程會暫時保留上一個已知 session，避免索引查詢視圖瞬間斷掉。

## 冷啟動 bootstrap

冷啟動由 `SessionStartupCoordinator.bootstrapSession()` 主導。

流程如下：

1. 先確認目前平台是否支援 vault runtime；目前非 Android 會進 `fatalError`。
2. `VaultRepository.initialize()` 準備目錄與快取 metadata。
3. 讀 `recovery.json`：
   - 若不存在，狀態進 `unlocked`，但 `session == null`，引導使用者先建立 Recovery Key。
4. 若已有 recovery metadata，檢查本機是否有 trusted device：
   - 沒有：進 `recoveryRequired`
   - 有：直接走 `unlock()`
5. trusted unlock 成功後進 `unlocked + session`

注意：

- 冷啟動時如果 trusted unlock 失敗，不一定都回 `locked`。
- 若判定是認證錯誤、索引不可讀、Keystore 資料失效、wrapped record 格式錯誤等永久性問題，流程可能清掉 trusted device 並進 `recoveryRequired`。
- 只有一般未知錯誤才會進 `fatalError`。

## Startup Phase 與 lifecycle 隔離

bootstrap 結束後，session 還會進入 `SessionStartupPhase`，用來隔離「冷啟動剛回到前景」與「真正背景回來」這兩件事。

| 階段 | 意義 |
|------|------|
| `bootstrapping` | bootstrap 進行中，不參與 lifecycle resume 解鎖 |
| `awaitingFirstBackground` | bootstrap 已結束，但尚未開始正式記錄背景事件 |
| `resumeUnlockArmed` | 已允許記錄真實背景，之後可參與 resumed 自動解鎖 |

實際行為：

1. bootstrap 結束時，phase 進入 `awaitingFirstBackground`
2. 若當下狀態是 `unlocked + session`，前景穩定約 **500ms** 後才會 `armLifecycleResumeUnlock()`
3. 只有進入 `resumeUnlockArmed` 之後，`paused` / `hidden` 才會被記成有效背景
4. 因此冷啟動後第一個 `resumed` 不會被誤判成「應該自動再解鎖一次」

## trusted device 實際在做什麼

trusted device 解鎖不是逐檔透過 Keystore 解 `LDJ2` 檔頭，而是先恢復 vault 層的 `recoveryWrapKey`。

trusted device 路徑需要兩部分同時存在：

- Android Keystore 內對應 slot
- `flutter_secure_storage` 內的 `WrappedRecoveryKeyRecord`

高層流程：

1. `VaultRepository.openTrustedSessionEnsuringKeystore()` 讀取 recovery metadata 與 wrapped record。
2. 先檢查目前 wrapped record / slot 是否符合現行 unlock mode 期待。
3. 若一致，直接 `openTrustedSession()`。
4. 若不一致，先做 Keystore migration / rewrap，再 purge 掉其他 inactive device keys。
5. 成功後開啟索引 `ensureIndexReady()`，並把 Keystore 保護後綴同步寫入索引 app value。
6. 最後產生 `UnlockedVaultSession(vaultId, recoveryWrapKey, trustedDevice, deviceSlotId)`。

這代表 trusted device 解鎖其實同時包含：

- 恢復 `recoveryWrapKey`
- 驗證 metadata 與 key 是否匹配
- 確認目前 Keystore 保護形態與使用者偏好一致
- 準備索引連線

檔案層 `LDJ2` header / recovery slot 細節請看 [加密格式.md](./加密格式.md)。

## Recovery Key 解鎖

`unlockWithRecovery()` 會先進 `AppSessionController`，再經 `VaultRecoveryService.unlockWithRecoveryKey(recoveryKey)` 進入底層 repository。

這條路徑不只是驗證 Recovery Key，還會順便把 trusted device 狀態補齊。可分成兩段看：

底層 repository 內會先做：

1. 用 `recovery.json` 的 KDF 參數衍生 `recoveryWrapKey`
2. 驗證這把 key 確實能打開 vault 樣本
3. 讀目前 unlock mode 偏好，並先做 unlock mode capability 檢查
4. 以對應 auth kind 建立或更新 trusted device wrapped record，並同步 wrapped recovery key
5. 把 Keystore 保護後綴寫回索引 app value，讓索引 metadata 與目前保護型態一致
6. 固定執行 `rebuildIndex()`，把 rewrap 後的 vault 狀態完整同步回索引

回到 application 層後會再做：

1. `ensureIndexReady()`，確認索引連線可直接沿用
2. `ensureKeystoreMatchesUnlockMode()`，確認 Keystore 保護型態與目前偏好一致
3. 將 session 狀態切成 `unlocked`

如果 Recovery Key 錯誤或驗證失敗，session 會進 `recoveryRequired`，並保留對應錯誤訊息。

## 背景逾時

背景逾時由 `SessionInactivityWatchdog` 控制。

目前規則：

- 預設 3 分鐘
- 可調成 `1 / 3 / 5 / 10` 分鐘
- 只有 `paused` / `hidden` 會開始計時
- `inactive` 不算背景逾時起點
- 回到 `resumed` 後不會立刻判定，而是先等約 **500ms** 前景穩定期

流程如下：

1. session 解鎖後會 `arm(timeout, onExpired)`
2. App 進 `paused` / `hidden` 時開始記 `backgroundSince`
3. `resumed` 後先取消背景 timer，再檢查是否已逾時
4. 若已逾時，觸發 `expireFromInactivity()`
5. 若尚未逾時，等待 500ms 前景穩定期
6. 穩定期內若狀態仍有效，就清掉背景紀錄，保留 session

逾時後的結果：

- state 變成 `locked`
- `lockReason = inactivity`
- 會先保存鎖定前路由，供解鎖成功後導回
- vault / 索引等已開資源會關閉
- `activeVaultSessionProvider` 不再提供 session

補充：

- `onPointerDown()` 在已解鎖時也會提早清掉背景追蹤，作為額外保險。
- `runSensitiveTask()` 與 `runBackgroundSafeTask()` 執行期間，watchdog 會暫時 disarm，避免長作業中途被背景逾時打斷。
- 若鎖定時仍有進行中的 sensitive task，真正的資源 cleanup 會延後到任務結束後才做。

## Resume 自動解鎖

回到前景後，是否會自動重新驗證，取決於三件事：

- 已經進入 `resumeUnlockArmed`
- 這個啟動週期內曾經有過有效背景事件
- `state.shouldUnlockOnResume == true`

其中 `shouldUnlockOnResume` 的定義是：

- `status == locked`
- `lockReason != manual`

也就是說，手動上鎖後不會因為 resumed 自動再打開。

實際流程由 `AppLifecycleSessionBridge` 控制：

1. App `resumed`
2. 先讓 watchdog 做前景判定
3. 若符合 resume unlock 條件，等待約 **350ms**
4. 呼叫 `unlock(source: lifecycleResume)`

不同模式的 resumed 行為：

- `none`：直接 trusted device unwrap，不跳系統驗證
- `deviceLock`：系統螢幕鎖驗證
- `biometric`：系統生物驗證，必要時可能 fallback 到螢幕鎖

若 resumed 觸發的驗證沒有成功：

- 使用者取消或驗證逾時：回到原本 `locked` 狀態，且保留原本 `lockReason`
- 驗證失敗：進 `locked + lockReason: authFailed`
- 生物驗證未登錄、裝置沒有憑證等：進 `locked` 並顯示對應訊息
- wrapped record 無效、Keystore key invalidated、索引不可讀等永久失敗：清除 trusted device 並進 `recoveryRequired`

## 手動上鎖與手動重驗證

使用者主動上鎖時：

- state 進 `locked + lockReason: manual`
- watchdog disarm
- 不會在同一次 resumed 內自動解鎖

之後若使用者主動按「重新驗證」，仍可再次走 `unlock()`。

設定頁在某些 `unlocking` 狀態下，也可以讓使用者取消目前驗證並改成手動重驗證；本質上同樣是把鎖定原因轉成 `manual`。

## Restore 後的 session 分流

restore 完成後不能直接讓一般 startup 與 restore startup 互相競爭，因此要走 `finishRestoreSession()`。

它會先：

- `beginPostRestoreStartup()`
- invalidate vault / index / settings 相關 provider
- 關閉舊資源

接著依情況分四條路：

1. 使用者有輸入備份的 Recovery Key：
   - 直接 `unlockWithRecovery()`
2. 若 restore 前有 live session，且 precheck 判定還可沿用：
   - 走 `resumeSessionAfterRestore(priorSession)`
   - 這會重用原本 session 內的 `recoveryWrapKey`
   - 不再要求使用者重新做裝置驗證
3. 若預期應該 trusted unlock，但前置條件已不成立：
   - 直接進 `recoveryRequired`
4. 其他情況：
   - 走 `bootstrapAfterRestore()`

另外：

- `resumeSessionAfterRestore()` 成功後會 `ensureIndexReady()` 並 `rebuildIndex()`
- 用 Recovery Key 還原成功後，會沿 `unlockWithRecovery()` 路徑完成 trusted-device rewrap 並固定 `rebuildIndex()`
- restore 結束時會清掉 route snapshot，並重新刷新首頁索引快取

## 切換解鎖模式

解鎖模式切換不是單純改一個偏好值，而是要同步調整 trusted device 保護。

`UnlockModeChangeService.apply()` 的實際順序是：

1. 讀出目前模式
2. 若目標模式沒變，直接成功
3. 用 `precheckUnlockModeChange()` 做 `migrate` 能力檢查
4. 呼叫 `VaultRepository.ensureKeystoreMatchesUnlockMode(session, targetMode: mode)`
5. 只有在 Keystore 同步成功後，才 `appLock.setUnlockMode(mode)`

重點：

- 切換時必須已有有效 `UnlockedVaultSession`
- `deviceLock` 需要 device credential
- `biometric` 在切換時需要 strong biometric 已登錄
- 切換過程若使用者取消系統驗證，偏好不會先寫入
- 成功後 session 可能帶回新的 `deviceSlotId`

## 什麼時候會退回 Recovery Key

trusted device 是便利入口，不是資料所有權本體。以下情況會退回 Recovery Key：

- 本機沒有 trusted device record
- wrapped record 與 Keystore slot 已失效
- Keystore key 被系統 invalidated
- trusted unlock 驗證後發現實際 key 無法打開 vault
- 還原後 `vaultId` 與既有 session 不一致
- 索引或樣本檔驗證顯示目前 trusted 狀態已不可信

一旦進 `recoveryRequired`，使用者就必須改走 Recovery Key 路徑，不能只靠 trusted device 繼續。
