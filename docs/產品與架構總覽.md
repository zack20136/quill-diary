# 產品與架構總覽

這份文件只保留**目前仍成立**的產品定位與架構決策，不再收錄大量早期比較、候選方案或拆工草稿。

## 1. 產品定位

QuillLockDiary 是一個：

> **本地優先、離線可用、以隱私與可搬移性為核心的 Android 日記 App。**

使用者看到的是一般日記體驗：

- 日期與標題
- 內文編輯
- 圖片附件
- 日曆、時間線、搜尋
- 本機備份與 Google Drive 備份

底層則是：

- 加密的 Markdown 條目
- 加密附件
- 可重建的 SQLite 索引
- Recovery Key 與受信任裝置解鎖機制

## 2. 穩定產品原則

### 2.1 使用者不需要理解底層格式

Markdown 是儲存格式，不是操作門檻。使用者不需要知道：

- YAML front matter
- `.md.enc` / `*.enc`
- `LDJ2` header
- `key slot`
- SQLite index

### 2.2 隱私優先

- 預設不需要帳號。
- 預設不把明文日記傳上雲端。
- Google Drive 只存放**加密備份**。
- Recovery Key 不完整落地成一般檔案。

### 2.3 可搬移

已解鎖 session 下可匯出：

```text
entries/**/*.md
assets/**/*
```

也就是說，App 內部雖然以加密 vault 儲存，但使用者仍能拿回一般 Markdown 與附件。

## 3. 穩定架構決策

### 3.1 只採用模式 A

固定採用：

```text
App 內部加密，匯出時才變成明文 Markdown
```

因此磁碟上的真實資料是：

- `vault/entries/**/*.md.enc`
- `vault/assets/**/*.enc`

### 3.2 每個檔案一把 `fileKey`

目前實作不採單一 `vaultKey` 解整包 vault，而是：

- 每份加密檔都有自己的 `fileKey`
- 同一把 `fileKey` 會同時被包進裝置槽與 Recovery 槽

這讓檔案隔離性、重包裝（rewrap）與災難復原都更單純。

### 3.3 索引是衍生資料，不是真相來源

SQLite 用於：

- 列表與搜尋
- 日期查詢
- 附件關聯

但真相來源仍是加密檔本身；索引損毀後可於解鎖狀態下重建。

### 3.4 備份是快照，不是同步

`.jbackup` 是某一時間點的 zip 快照，不是雙向同步資料夾，也不是多裝置同步機制。

## 4. 支援範圍

| 項目 | 狀態 |
|------|------|
| Android | 正式支援 |
| `LDJ2` | 正式支援 |
| Recovery Key v2+ | 正式支援 |
| 舊版 vault migration | 不支援 |
| Web / Desktop 資料操作 | 不支援 |

## 5. 威脅模型邊界

本專案主要要防：

- 裝置遺失後直接讀取 app 檔案
- 備份外洩後直接看到明文
- Android Keystore 失效後仍能以 Recovery Key 救援

本專案**不承諾**防：

- 已解鎖裝置上的惡意鍵盤、螢幕錄影或截圖外洩
- root 且執行中遭植入惡意程式的裝置
- 使用者自行外洩 Recovery Key
