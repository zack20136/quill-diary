# 解鎖與會話

這份文件整理 Quill Diary 的解鎖與 session 行為，包括解鎖模式、可信裝置流程、背景逾時、resume 行為與設定頁切換。

它只講 session 與解鎖體驗，不展開加密檔案格式本身。

## 這份機制在做什麼

Quill Diary 不是只做一次解鎖就結束，而是維持一段「目前可讀寫日記庫」的 session。

這份機制主要解決幾件事：

- App 什麼時候算已解鎖
- 使用者回到前景時是否要重新驗證
- 可信裝置如何提供較順的重新進入體驗
- 什麼情況下必須退回復原金鑰路徑

## 兩種解鎖模式

目前有兩種可信裝置解鎖模式。

| 模式 | storage | 回前景驗證方式 | Keystore 槽 |
|------|---------|----------------|-------------|
| 裝置螢幕鎖 | `deviceLock` | 系統螢幕鎖對話框 | `deviceCredential` |
| 生物驗證 | `biometric` | 系統指紋或臉部驗證 | `biometric` |

補充：

- 兩種模式都要求裝置本身已設定螢幕鎖
- 生物驗證模式還要求至少有一種生物辨識已登錄
- 即使選了生物驗證，系統驗證流程內仍可能允許使用螢幕鎖作為替代方式

## Session 是什麼

session 代表目前 App 是否持有可用的解鎖狀態，能夠存取日記庫、草稿與搜尋索引。

高層來看：

- 有效 session 存在時，可讀寫正式日記、草稿與索引
- session 清掉後，App 會回到鎖定狀態
- 鎖定狀態下不能直接存取這些敏感資料

## 核心元件分工

| 元件 | 職責 |
|------|------|
| `AppSessionController` | 管理 session 狀態、`resumeAction`，並呼叫 `VaultRepository.openTrustedSession` |
| `SessionUnlockCoordinator` | 監聽 `resumeAction`，在需要時自動觸發 `unlock()` |
| `AppLockService` | 保存解鎖模式，並檢查裝置是否支援對應的系統驗證能力 |
| `DeviceKeyManager` | 管理 Android Keystore 的 wrap / unwrap |
| `VaultRepository` | 建立可信裝置 session，並確認 Keystore 狀態是否符合目前解鎖模式 |

這幾個元件一起決定目前是否能以可信裝置路徑重新進入日記庫。

## Resume 行為

App 回到前景時，不一定直接可用，可能需要依 `ResumeUnlockAction` 做後續處理。

目前主要行為是：

| 值 | 行為 |
|----|------|
| `keystoreUnlock` | 呼叫 `unlock()`，跳出系統驗證對話框 |

如果使用者取消或驗證失敗：

- App 仍維持 `locked`
- 使用者可手動按「重新驗證」再次觸發

## 背景逾時

session 不會永久保持有效。當 App 進背景過久，再回來時就必須重新驗證。

- 預設門檻為 5 分鐘
- 逾時策略由 `SessionTimeoutPolicy` 控制
- 一旦逾時，既有 session 會被清除
- 回到前景後，必須重新完成系統驗證才能恢復可信 session

這個行為的重點是：可信裝置只是在一段有效時間內降低重新進入成本，不是永久免驗證。

## 切換解鎖模式

設定頁可以在兩種可信裝置解鎖模式之間切換，但不是任意切。

切換流程重點：

1. 必須已建立復原金鑰
2. 切換前先檢查裝置是否可用對應的驗證能力
3. 使用者要先通過系統驗證
4. `ensureKeystoreMatchesUnlockMode` 完成後，才會寫入新偏好

這代表模式切換不只是 UI 選項切換，還包含底層 Keystore 狀態調整。

## 什麼時候會退回復原金鑰

可信裝置解鎖只是便利路徑，不是最終的資料所有權依據。

以下情況會讓使用者不能只靠可信裝置繼續使用，而必須退回復原金鑰路徑：

- 可信裝置狀態失效
- Keystore 狀態與目前解鎖模式不一致
- 還原後偵測到目前可信狀態與日記庫不再匹配
- 進入 `recoveryRequired` 狀態

這也是為什麼復原金鑰始終是必要前提，而不是可選附加功能。
