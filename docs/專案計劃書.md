## 1. 專案定位

本專案是一款：

> **本地優先、離線可用、重視隱私、安全加密、可匯出 Markdown 的 Android 日記 App。**

設計方向參考 Easy Diary 的簡單日記體驗，例如：

- 日曆
- 時間線
- 圖片附件
- PIN / 指紋鎖
- Google Drive 備份
- 不需要註冊帳號即可使用

但本專案的資料格式與安全架構會更接近：

> **加密版 Markdown vault。**

本專案的差異化：

```text
Easy Diary 的好寫
+ Obsidian / Joplin 類 Markdown 可搬移性
+ App 內部全加密
+ 高熵 Recovery Key 救援
+ Google Drive 只存加密備份
```

---

## 2. 核心產品原則

### 2.1 使用者體驗原則

使用者看到的是一個簡單日記 App，而不是 Markdown 工具。

```text
使用者看到：
  標題
  日期
  心情
  Tag
  正文編輯器
  圖片插入
  日曆
  時間線
  搜尋
  備份 / 還原

App 內部：
  .md.enc
  assets/*.enc
  index.sqlite
  recovery key slot
```

核心原則：

> **Markdown 是儲存格式，不是使用者負擔。**

使用者不需要理解：

- YAML front matter
- Markdown 圖片路徑
- `.md.enc`
- 加密 header
- key slot
- SQLite index

這些都由 App 處理。

---

### 2.2 隱私原則

```text
1. 預設不需要帳號
2. 預設不連網
3. 日記預設只存在本機
4. 本機檔案不是明文
5. Google Drive 只存加密備份
6. 使用者可匯出明文 Markdown vault
7. `.jbackup` 是系統備份封包，不是同步資料夾
```

日記屬於高度私密資料，因此：

- 不把明文日記傳到雲端
- 不把日記內容寫入 log
- 不把日記內容放入 crash report
- 不把完整 Recovery Key 存在手機內
- 不將 Google Drive 當成資料保護機制，只把它當成加密檔案儲存空間

---

### 2.3 可搬移原則

使用者不能被 App 綁死。

匯出後應該是：

```text
MyDiaryExport/
  manifest.json
  entries/
    2026/
      05/
        2026-05-13-今天散步.md
  assets/
    2026/
      05/
        att_01HXZ9AFQ2.webp
```

匯出的 Markdown 應盡量符合 CommonMark 基礎語法，避免使用太多 App 專屬格式。

## 3. 最終架構決策

### 3.1 採用模式 A

固定採用：

```text
模式 A：App 內部加密，匯出時才變 Markdown
```

App 內部：

```text
vault/
  manifest.json.enc

  entries/
    2026/
      05/
        jrn_20260513_203012_a8f3.md.enc

  assets/
    2026/
      05/
        att_01HXZ9AFQ2.webp.enc

  index/
    journal_index.sqlite
```

匯出時：

```text
MyDiaryExport/
  manifest.json
  entries/
    2026/
      05/
        2026-05-13-今天散步.md
  assets/
    2026/
      05/
        att_01HXZ9AFQ2.webp
```

---

### 3.2 不使用 Supabase 作為核心

因為本專案需求是：

```text
完全本地端
頂多 Google Drive 備份
平常不連網
重視安全
可搬移
```

所以 Supabase 不適合作為核心資料層。

Supabase 適合：

- 雲端同步
- 帳號系統
- 多人資料
- 後端資料庫
- 即時資料

但這個 App 的核心應該是：

```text
本地 vault
+ 本地加密
+ 本地索引
+ 手動加密備份
```

---

### 3.3 單一真實資料來源

固定採用：

```text
entries/*.md.enc 與 assets/*.enc
才是真實資料來源
```

以下都屬於可重建的衍生資料：

```text
SQLite index
manifest.json.enc
backup manifest
FTS 索引
```

這個決策代表：

- index 損壞時可由 vault 重建
- 備份可包含 index snapshot，但不能依賴它才能還原
- 匯入與還原時一律以加密檔案實體為準

---

### 3.4 不引入 vaultKey

第一版固定採用：

```text
每個日記檔案一把 fileKey
每個附件檔案一把 fileKey
不引入可解開整包 vault 的單一 vaultKey
```

理由：

- 單篇救援更直接
- Recovery Key 輪替只需重包裝 fileKey
- 單一金鑰洩漏不會導致整包資料失守
- 後續多裝置 device slot 也較自然

---

## 4. Recovery Key 最終設計

### 4.1 Recovery Key 採高熵隨機產生

固定採用：

```text
Recovery Key = App 產生的高熵隨機金鑰
```

不把「使用者自己輸入的普通密碼」當主要救援方式。

原因：

| 方案 | 評估 |
|---|---|
| 高熵隨機 Recovery Key | 最安全，抗猜測能力高 |
| 使用者自訂密碼 | 容易太弱，需要很重的 KDF 保護 |
| PIN | 只能當 App Lock，不適合保護救援資料 |
| Android Keystore | 適合本機，不適合跨裝置搬移 |

Recovery Key 可以顯示成使用者可抄寫的格式，例如：

```text
LDJ-8K4F-2Q9M-X7CP-R6TA-H3JD-K92L-PQ6V
```

建議使用分段代碼格式，因為：

```text
1. 好輸入
2. 好檢查
3. 好分段顯示
4. 可以加 checksum
```

---

### 4.2 Recovery Key 的角色

Recovery Key 不是登入密碼，而是：

> **跨裝置救援用的總鑰匙。**

用途：

```text
1. 換手機時解鎖舊 .md.enc
2. 從 Google Drive 還原 .jbackup
3. 解鎖單篇從其他裝置抓來的 .md.enc
4. 匯入舊 vault
5. 在新裝置重新建立本機 device slot
```

---

### 4.3 Recovery Key 不明文儲存在手機

App 可以存：

```text
recovery_enabled = true
recovery_key_hint = "PQ6V"
recovery_key_created_at = "2026-05-13T20:00:00+08:00"
recovery_key_version = 1
```

App 不存：

```text
完整 Recovery Key
```

---

### 4.4 Recovery Key 啟用流程

第一次設定安全性時：

```text
1. 使用者設定 PIN / 生物辨識
2. App 產生高熵 Recovery Key
3. 顯示 Recovery Key
4. 要求使用者抄下來
5. 要求使用者重新輸入一次確認
6. 啟用 recovery slot
7. 才完成安全設定並進入 App
```

畫面文案建議：

```text
這組 Recovery Key 可以解開你的日記。
如果你換手機、重裝 App，或本機金鑰失效，需要它才能還原資料。

請抄在紙上或存放在安全的密碼管理器。
不要截圖。
不要傳給別人。
App 無法幫你找回這組 Recovery Key。
```

## 5. 加密架構

### 5.1 採用 Envelope Encryption

固定採用：

```text
信封式加密 Envelope Encryption
```

概念：

```text
每篇日記 / 每張圖片：
  使用獨立 fileKey 加密內容

fileKey：
  被 device key slot 包裝
  也被 recovery key slot 包裝
```

這樣做的好處：

| 好處 | 說明 |
|---|---|
| 可單篇救援 | 只拿到某篇 `.md.enc` 也能用 Recovery Key 解開 |
| 可整包轉移 | 整個 vault 可批次匯入 |
| 可換 Recovery Key | 只需重新包裝 fileKey，不必重加密正文 |
| 可多裝置 | 每台裝置可以有自己的 device slot |
| 可降低損毀影響 | 單檔損壞不一定影響整個 vault |

---

### 5.2 金鑰層級

```text
Recovery Key
  ↓ KDF / key derivation
recoveryWrappingKey
  ↓ unwrap
fileKey
  ↓ decrypt
.md.enc / .webp.enc
```

日常使用：

```text
Android Keystore / App Lock
  ↓
deviceWrappingKey
  ↓
unwrap fileKey
  ↓
解密 .md.enc
```

救援 / 轉移：

```text
Recovery Key
  ↓
recoveryWrappingKey
  ↓
unwrap fileKey
  ↓
解密 .md.enc
```

Android Keystore 適合保護本機金鑰，但不應作為唯一的跨裝置救援依據，因為 Keystore 金鑰通常是裝置綁定、不可匯出的。

---

### 5.3 KDF 設計

即使 Recovery Key 是高熵隨機產生，仍建議經過 KDF 轉成實際 wrapping key。

建議：

```text
KDF：Argon2id
輸入：Recovery Key
salt：每個 vault 或每個檔案獨立 salt
輸出：recoveryWrappingKey
```

若 Flutter 套件支援度不足，MVP 可先抽象成 `KdfService`，底層實作可逐步替換。

```text
abstract class KdfService {
  Future<Uint8List> deriveRecoveryWrappingKey(
    String recoveryKey,
    Uint8List salt,
  );
}
```

## 6. `.md.enc` 檔案格式

### 6.1 不把加密檔命名成 `.md`

App 內部不要叫：

```text
2026-05-13.md
```

應該叫：

```text
jrn_20260513_203012_a8f3.md.enc
```

匯出時才生成：

```text
2026-05-13-今天散步.md
```

---

### 6.2 單篇 `.md.enc` 建議格式

```text
Magic Header:
  LDJ1

Header Length:
  uint32

JSON Header:
{
  "schema_version": 1,
  "file_id": "jrn_20260513_203012_a8f3",
  "vault_id": "vlt_01HXYZ...",
  "content_type": "text/markdown",
  "created_at": "2026-05-13T20:30:12+08:00",
  "updated_at": "2026-05-13T21:02:44+08:00",

  "crypto": {
    "cipher": "xchacha20-poly1305 或 aes-256-gcm",
    "nonce": "base64...",
    "aad": "base64..."
  },

  "key_slots": [
    {
      "type": "device",
      "id": "android_keystore_device_01",
      "wrapped_file_key": "base64..."
    },
    {
      "type": "recovery",
      "id": "recovery_key_v1",
      "kdf": {
        "name": "argon2id",
        "salt": "base64...",
        "memory_cost": 65536,
        "time_cost": 3,
        "parallelism": 1
      },
      "wrapped_file_key": "base64..."
    }
  ]
}

Ciphertext:
  encrypted markdown body
```

---

### 6.3 為什麼每個檔案都要有 recovery slot？

因為核心需求是：

> 怕 `.md` 被鎖起來，希望能從其他裝置抓來 `.md.enc` 後用 Recovery Key 解開。

因此每個 `.md.enc` 至少要有：

```text
device slot
recovery slot
```

這樣就算：

```text
舊手機壞掉
App 重裝
Android Keystore 失效
從 Google Drive 抓回舊檔案
從其他裝置拿到單篇 .md.enc
```

仍能用 Recovery Key 解開。

## 7. Vault 格式

### 7.1 App 內部 vault

```text
vault/
  manifest.json.enc
  recovery.json

  entries/
    2026/
      05/
        jrn_20260513_203012_a8f3.md.enc

  assets/
    2026/
      05/
        att_01HXZ9AFQ2.webp.enc

  index/
    journal_index.sqlite
```

---

### 7.2 recovery.json

`recovery.json` 可以保存非敏感資訊，不存完整 Recovery Key。

```json
{
  "schema_version": 1,
  "vault_id": "vlt_01HXYZ...",
  "recovery_enabled": true,
  "recovery_key_version": 1,
  "recovery_key_hint": "PQ6V",
  "created_at": "2026-05-13T20:00:00+08:00",
  "kdf": {
    "name": "argon2id",
    "purpose": "recovery_wrapping"
  }
}
```

---

### 7.3 manifest.json.enc

`manifest.json` 建議加密，避免暴露使用者日記數量、日期範圍、tag 等隱私資訊。

```json
{
  "schema_version": 1,
  "vault_id": "vlt_01HXYZ...",
  "entry_count": 128,
  "asset_count": 342,
  "oldest_entry_date": "2025-01-01",
  "newest_entry_date": "2026-05-13",
  "app_version": "1.0.0"
}
```

補充原則：

```text
manifest.json.enc 不是唯一真相來源
如果 manifest 損壞，可由 entries / assets 重建
```

## 11. SQLite 索引設計

Markdown 是真實資料，但 App 不應每次都掃所有 `.md.enc`。日曆、搜尋、tag 應由 SQLite index 提供。

SQLite 的角色固定為：

```text
查詢加速層
可重建衍生層
不是唯一真相來源
```

### 11.1 entries_index

```sql
CREATE TABLE entries_index (
  id TEXT PRIMARY KEY,
  vault_id TEXT NOT NULL,
  file_path TEXT NOT NULL,
  title TEXT,
  title_normalized TEXT,
  preview_text TEXT,
  date TEXT NOT NULL,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  timezone TEXT,
  mood TEXT,
  weather TEXT,
  location_text TEXT,
  word_count INTEGER NOT NULL DEFAULT 0,
  char_count INTEGER NOT NULL DEFAULT 0,
  attachment_count INTEGER NOT NULL DEFAULT 0,
  has_attachments INTEGER NOT NULL DEFAULT 0,
  is_favorite INTEGER NOT NULL DEFAULT 0,
  is_deleted INTEGER NOT NULL DEFAULT 0,
  deleted_at TEXT,
  schema_version INTEGER NOT NULL DEFAULT 1,
  encrypted_file_size INTEGER,
  encrypted_file_mtime TEXT,
  content_hash TEXT
);
```

### 11.2 entry_tags

```sql
CREATE TABLE entry_tags (
  entry_id TEXT NOT NULL,
  tag TEXT NOT NULL,
  tag_normalized TEXT NOT NULL,
  PRIMARY KEY (entry_id, tag_normalized)
);
```

### 11.3 entry_attachments

```sql
CREATE TABLE entry_attachments (
  id TEXT PRIMARY KEY,
  entry_id TEXT NOT NULL,
  file_path TEXT NOT NULL,
  mime_type TEXT NOT NULL,
  safe_filename TEXT NOT NULL,
  width INTEGER,
  height INTEGER,
  byte_size INTEGER NOT NULL,
  sha256 TEXT NOT NULL,
  created_at TEXT NOT NULL,
  is_deleted INTEGER NOT NULL DEFAULT 0
);
```

### 11.4 entries_fts

```sql
CREATE VIRTUAL TABLE entries_fts USING fts5(
  entry_id,
  title,
  body,
  tags,
  preview_text
);
```

### 11.5 app_kv / backup_history

```sql
CREATE TABLE app_kv (
  key TEXT PRIMARY KEY,
  value TEXT NOT NULL,
  updated_at TEXT NOT NULL
);

CREATE TABLE backup_history (
  backup_id TEXT PRIMARY KEY,
  provider TEXT NOT NULL,
  remote_file_id TEXT,
  created_at TEXT NOT NULL,
  status TEXT NOT NULL,
  entry_count INTEGER NOT NULL,
  asset_count INTEGER NOT NULL,
  byte_size INTEGER,
  error_code TEXT
);
```

### 11.6 日曆資料

```sql
SELECT date, COUNT(*)
FROM entries_index
WHERE is_deleted = 0
GROUP BY date;
```

## 12. Google Drive 備份設計

### 12.1 備份原則

```text
1. 不做即時同步
2. 不做背景偷偷上傳
3. 使用者手動備份
4. Google Drive 只存加密檔
5. Recovery Key 可用於還原
```

Google Drive `appDataFolder` 適合存 App 專用備份，但本專案仍只上傳加密備份。

---

### 12.2 備份檔格式

```text
journal-backup-2026-05-13-2030.jbackup
```

內部：

```text
backup_manifest.json
recovery.json
entries/*.md.enc
assets/*.webp.enc
index snapshot
```

補充定義：

```text
.jbackup 是不可變快照
不是同步資料集
不是一般使用者手動維護的資料夾格式
```

整個 `.jbackup` 也可以再做一層加密與完整性驗證。

---

### 12.3 備份流程

```text
1. 使用者進入設定
2. 點擊「備份到 Google Drive」
3. App 要求再次解鎖
4. 產生 .jbackup
5. 確認 recovery slot 存在
6. 上傳到 Google Drive appDataFolder
7. 顯示備份時間與備份大小
```

---

### 12.4 還原流程

```text
1. 使用者安裝新 App
2. 選擇「從 Google Drive 還原」
3. 登入 Google
4. 下載 .jbackup
5. 輸入 Recovery Key
6. 解開 recovery slot
7. 解密 entries / assets
8. 匯入新 vault
9. 在新裝置建立 device slot
10. 重建 SQLite index
```

## 13. Recovery Mode 功能

這是本專案的核心安全與搬移功能。

### 13.1 功能入口

```text
設定 → 資料救援
```

提供三個功能：

```text
1. 從 Google Drive 還原
2. 匯入加密 vault
3. 解鎖單篇 .md.enc
```

---

### 13.2 解鎖單篇 `.md.enc`

流程：

```text
1. 使用者選擇 .md.enc
2. App 讀取 header
3. App 檢查是否有 recovery slot
4. 使用者輸入 Recovery Key
5. App 推導 recoveryWrappingKey
6. 解開 wrapped_file_key
7. 解密 Markdown
8. 顯示預覽
9. 使用者可選擇：
   - 匯入目前 vault
   - 匯出成 .md
   - 只預覽不保存
```

---

### 13.3 匯入加密 vault

流程：

```text
1. 使用者選擇 vault 資料夾或 .jbackup
2. App 掃描 entries / assets
3. 使用者輸入 Recovery Key
4. App 批次解密
5. 匯入目前裝置
6. 建立新的 device slot
7. 重建 index
```

---

### 13.4 更換 Recovery Key

流程：

```text
1. 使用者通過本機 App Lock
2. App 產生新的高熵 Recovery Key
3. 顯示並要求重新輸入確認
4. App 重新包裝所有 fileKey
5. 移除舊 recovery slot
6. 完成輪替
```

重點：

> 更換 Recovery Key 時不需要重加密整篇 Markdown，只需要重新包裝 fileKey。這是信封式加密的主要優點。

## 20. 測試計劃

### 20.1 加密測試

```text
1. 正確 Recovery Key 可解密
2. 錯誤 Recovery Key 無法解密
3. 舊 device slot 不可用時，recovery slot 可用
4. 更換 Recovery Key 後舊 key 失效
5. 單篇 .md.enc 可獨立救援
6. 圖片 .webp.enc 可獨立救援
```

---

### 20.2 備份還原測試

```text
1. 備份後刪除 App，重新安裝可還原
2. 換裝置可還原
3. Google Drive 上看不到明文內容
4. recovery.json 不包含完整 Recovery Key
5. 還原後可重建 SQLite index
```

---

### 20.3 Markdown 匯出測試

```text
1. 匯出的 .md 可用一般文字編輯器打開
2. 圖片相對路徑正確
3. front matter 正確
4. tag / mood / 日期正確
5. 可匯入 Obsidian 類 vault
```

---

### 20.4 UX 測試

```text
1. 使用者是否理解 Recovery Key 的重要性
2. 使用者是否能成功抄寫並確認 Recovery Key
3. 使用者是否能不懂 Markdown 也寫日記
4. 使用者是否能找到日曆與搜尋功能
5. 使用者是否能完成備份與還原
```

## 21. 風險與對策

| 風險 | 說明 | 對策 |
|---|---|---|
| 使用者弄丟 Recovery Key | 換手機後可能無法還原 | onboarding 強提醒、確認輸入、顯示備份狀態 |
| Recovery Key 被別人取得 | 可能解開日記 | 防截圖、警告文案、不明文儲存 |
| 加密格式設計錯誤 | 導致無法解密 | 先做 prototype + 測試向量 |
| Android Keystore 失效 | 本機無法日常解鎖 | recovery slot 可救援 |
| SQLite index 損壞 | 日曆搜尋失效 | 可由解密後 Markdown 重建 |
| Google Drive 備份失敗 | 使用者誤以為已備份 | 明確顯示最後成功備份時間 |
| Markdown 編輯門檻高 | 一般使用者不會語法 | TextField + toolbar + 預覽 |
| 圖片備份太大 | 備份慢 | 壓縮、顯示大小、可選是否備份原圖 |

## 23. 最終摘要

最終技術方向：

```text
Flutter
+ Markdown-friendly editor
+ .md.enc / assets.enc
+ SQLite index + FTS5
+ Android Keystore
+ 高熵隨機 Recovery Key
+ Envelope Encryption key slots
+ Google Drive appDataFolder encrypted backup
+ Markdown vault export
```

最重要的架構原則：

> **每篇 `.md.enc` 都不能只綁死在單一手機。它必須同時支援 device slot 與 recovery slot。**

一句話總結：

> 這個 App 要做的不是單純日記，也不是單純 Markdown 編輯器，而是「可救援、可搬移、可長期保存的本地加密日記系統」。
