# 實作與資料設計

這份文件整理目前程式裡**已固定下來**的資料模型、路徑慣例與實作邊界，避免長篇文件裡混雜過多「建議」「未來可」或過期方案。

## 1. 核心資料模型

### 1.1 `DiaryEntry`

代表一篇日記的業務資料，核心欄位包括：

- `id`
- `vaultId`
- `title`
- `date`
- `createdAt`
- `updatedAt`
- `tags`
- `mood`
- `markdownBody`
- `attachmentIds`
- `isDeleted`

正文不直接保存 front matter；front matter 只在 encode / decode Markdown 文件時生成。

### 1.2 `AssetAttachment`

代表一個附件，核心欄位包括：

- `id`
- `entryId`
- `mimeType`
- `originalFilename`
- `safeFilename`
- `width` / `height`
- `byteSize`
- `createdAt`
- `sha256`

其中 `safeFilename` 是匯出與 vault 中實際使用的穩定檔名。

### 1.3 `RecoveryMetadata`

對應 `vault/recovery.json`，核心資訊為：

- `vaultId`
- `recoveryEnabled`
- `recoveryKeyVersion`
- `recoveryKeyHint`
- `createdAt`
- `kdf`

`kdf` 目前固定為 `argon2id`，並帶 salt、memory、parallelism、iterations、hash length。

### 1.4 `UnlockedVaultSession`

解鎖後帶著的 session 包含：

- `vaultId`
- `trustedDevice`
- `recoveryWrapKey`
- `deviceSlotId`

crypto 層會把它再整理成 `DecryptionContext` 使用。

## 2. Markdown 與 front matter

單篇日記在加密前，會先被編碼成：

```text
YAML front matter
+
Markdown body
```

目前 front matter 至少會包含：

- `id`
- `title`
- `date`
- `created_at`
- `updated_at`
- `tags`
- `mood`
- `attachments`
- `schema_version`

附件在 front matter 裡記錄的是匯出路徑形式，例如：

```text
../assets/2026/05/example.webp
```

## 3. 路徑與目錄慣例

本機根目錄為：

```text
ApplicationSupportDirectory/quill_lock_diary/
```

主要子目錄如下：

| 路徑 | 說明 |
|------|------|
| `vault/recovery.json` | Recovery metadata 與 KDF 參數 |
| `vault/manifest.json.enc` | 加密 manifest |
| `vault/entries/<year>/<month>/<entryId>.md.enc` | 加密日記 |
| `vault/assets/<year>/<month>/<assetId>.<ext>.enc` | 加密附件 |
| `vault/index/journal_index.sqlite` | SQLite 索引 |
| `exports/` | 明文匯出 |
| `backups/` | `.jbackup` 備份快照 |

目錄分層以 `date.year` / `date.month` 為主，而不是 `createdAt`。

## 4. 加密與寫入邊界

本專案目前的穩定做法是：

- **每檔一把 `fileKey`**
- 內容加密演算法固定為 `aes-256-gcm`
- `headerBytes` 作為內容加密的 AAD
- 同時保留 `device` 與 `recovery` 兩種 key slot

寫入時採：

1. 先產生加密檔內容
2. 原子寫入 `.enc`
3. 再更新 SQLite 與搜尋索引
4. 最後重寫 `manifest.json.enc`

這代表：

- 真實資料先落地
- 索引永遠晚於真實資料
- App 中斷後，理論上仍可由加密檔重建索引

## 5. 刪除與重建策略

### 5.1 刪除

目前日記刪除採**軟刪除**路徑：

- 先把 `isDeleted` 寫回條目
- 再標記索引狀態

### 5.2 重建

`rebuildIndex` 會：

1. 重建 SQLite 結構
2. 掃描 `vault/entries/**/*.md.enc`
3. 逐篇解密、decode、回填 entry / 搜尋 / 附件關聯

因此重建需要**可用的 session**，不是單純拿到檔案就能完成。

## 6. Session 與鎖定狀態

### 6.1 狀態列舉

目前 session 狀態分為：

- `uninitialized`
- `locked`
- `unlocking`
- `unlocked`
- `recoveryRequired`
- `fatalError`

### 6.2 Recovery Key 試解密（`_verifyRecoveryKey`）

- 有 **`manifest.json.enc`**：對其做 recovery-only 試解密。
- **無 manifest**：在 `vault/entries`、`vault/assets` 下尋找 `*.enc`，優先所有 `.md.enc`（排序後）、再其他 `.enc`；對可 parse 的第一個成功案例即通過。**MAC 認證失敗**視為 Recovery Key 錯誤。若連一個可用的加密檔都沒有，則略過試解密。
- **`unlockWithRecoveryKey`**：`rewrap` **完成後** 才呼叫 **`_storeWrappedRecoveryKey`**，再 `rebuildIndex`。

### 6.3 `openTrustedSession`：兩種失敗分開處理

| 情境 | 行為 |
|------|------|
| `unwrapWithDeviceKey`（讀 trusted 保存的 wrapping key）失敗 | `clearTrustedKey`，請使用者用 Recovery Key 重新註冊 |
| `_resumeRewrapIfNeeded`／檔案 rewrap 失敗 | **不**清除 trusted；拋出可讀錯誤，`rewrap_in_progress` 可保留供下次繼續 |

### 6.4 啟動時大致流程

1. 初始化 repository
2. 讀 `recovery.json`
3. 若未建立 Recovery Key，進入可建立狀態
4. 若沒有 trusted device，要求輸入 Recovery Key
5. 若有 trusted device，嘗試 `openTrustedSession`
6. 若系統層 app lock 仍在，狀態為 `locked`

## 7. 備份、還原、匯出

### 7.1 備份

- `createBackupSnapshot` 會把整個 `vault/` 打成 `.jbackup`
- 備份內容仍是加密檔，不包含明文日記

### 7.2 還原

- `restoreBackup` 先解到暫存，再覆蓋 `vault/`
- 還原後需重新進入 session，必要時重建索引

### 7.3 匯出

- `exportMarkdownVault` 只在已解鎖 session 下執行
- 會把 entry 解密成 `.md`
- 也會把附件解密後輸出到 `exports/assets/...`
