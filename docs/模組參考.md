# 模組參考

主要型別、Repository 與 Provider 速查。

## 型別

### `UnlockedVaultSession`

代表目前已解鎖的日記庫 session，包含 `vaultId`、`trustedDevice`、`recoveryWrapKey`、`deviceSlotId`。

### `RecoveryMetadata`

對應 `vault/recovery.json`，保存日記庫 id、建立時間、復原金鑰 hint 與 Argon2id 參數。

### `PendingAttachment`

UI 已選取但尚未寫入日記庫的附件。

### `ResumeUnlockAction`

`autoTrusted` | `keystoreUnlock` | `deviceCredentialFallback`；見 [解鎖與會話.md](./解鎖與會話.md)。

## 模組

### `VaultRepository`

| 職責 |
|------|
| 日記庫初始化 |
| 本機受信任 session 還原（含 `deviceCredentialFallback`） |
| 復原金鑰設定與解鎖 |
| entry / asset 讀寫 |
| manifest 驗證 |
| 索引同步與 rewrap 協調 |
| `syncDeviceCredentialBackupWrapIfNeeded`、`ensureKeystoreMatchesUnlockMode` |

### `DeviceKeyManager`

| 職責 |
|------|
| 與 Android Keystore 溝通（plain / deviceCredential / biometric） |
| 管理本機受信任裝置 key |
| 包裝 / 解開 recovery wrapping key |
| 生物模式 credential 備份 wrap 讀寫 |

### `AppLockService`

| 職責 |
|------|
| 持久化 `AppUnlockMode`（`app_lock_store.json`） |
| `canUseDeviceCredential()`（螢幕鎖是否可用） |
| `keystoreAuthKindForCurrentMode()` |

### `IndexDatabaseManager`

| 職責 |
|------|
| 依 session 開啟索引資料庫 |
| 關閉、刪除、重建 index 相關檔案 |

### `TagStylesStore`

| 職責 |
|------|
| 讀寫 `vault/tag_styles.json` 中的標籤 accent 顏色 |
| 備份管線經 `VaultArchiveIo` 納入 `.jbackup` |
| 與索引資料庫 `tag_styles` 表同步；詳見 [索引資料庫.md](./索引資料庫.md) |

### `AppSessionController`

| 職責 |
|------|
| 啟動流程狀態 |
| 解鎖、鎖定、timeout、背景恢復 |
| 將本機受信任狀態錯誤收斂為 `locked` 或 `recoveryRequired` |
| 設定 `resumeAction`（含生物失敗 → 螢幕鎖備援） |

### `SessionUnlockCoordinator`

| 職責 |
|------|
| 監聽 `resumeAction`，在 home / settings 自動觸發 `unlock()` |
| 處理 `deviceCredentialFallback`（不再使用 App PIN 對話框） |

## Provider

| Provider | 用途 |
|----------|------|
| `appSessionProvider` | 執行期 session 互動狀態 |
| `appStartupProvider` | 啟動一次性初始化 |
| `effectiveAppSessionProvider` | startup 與 local 狀態合併 |
| `activeVaultSessionProvider` | 僅在 `unlocked` 時暴露 session |
| `shared/providers/core_providers.dart` | app-wide 基礎依賴（含 `appLockServiceProvider`） |

feature provider 只管理該 feature 的查詢、畫面狀態與互動。

## 設計原則

- 跨 feature 的通用 UI 與工具才放到 `shared/`
- 複雜流程寫在 repository / coordinator，不建立只轉呼叫一次的 use case
- 裝置驗證一律走系統 Keystore 對話框，不在 App 內實作第二套 PIN
