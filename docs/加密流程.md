# 加密流程（LDJ2）

說明文稿與程式邏輯如何從「可讀資料」寫入磁碟上的 **`.md.enc` / `*.enc`**。實作主檔：`lib/infrastructure/crypto/crypto_service.dart`。

## 流程圖

```mermaid
flowchart TD
    A[開始：markdown 或 bytes] --> B[產生隨機 fileKey]
    B --> C[DeviceKeyManager.wrapWithDeviceKey]
    B --> D[用 recoveryWrapKey 包裝 fileKey]
    C --> E[建立 device slot]
    D --> F[建立 recovery slot]
    E --> G[組合 EncryptedDocumentHeader]
    F --> G
    G --> H[header 轉成 headerBytes]
    H --> I[以 fileKey + nonce + AAD=headerBytes 做 AES-256-GCM]
    I --> J[輸出 LDJ2 檔案<br/>magic + header_len + headerBytes + ciphertext||tag]
    J --> K[Repository 原子寫入 .enc]
    K --> L[更新 SQLite 與 manifest.json.enc]
```

## 1. 名詞對照

| 名詞 | 意義 |
|------|------|
| **Recovery Key** | 使用者保存的高熵字串。**不會**直接拿去 encrypt 正文。 |
| **recoveryWrapKey** | 透過 Argon2id（參數在 `recovery.json` 內嵌的 `KdfDescriptor`）從 Recovery Key **導出**的 32 bytes，只用來包住每個檔案的 `fileKey`（recovery 槽）。 |
| **fileKey** | **每個加密檔案一把**（AES-256 randomly generated），encrypt 正文／二進位內容。 |
| **裝置槽（device slot）** | 用 Android Keystore 相關橋接，把同一個 `fileKey` 再做一次 AES-GCM 包裝；日常解鎖走這一路徑較省事。 |
| **Recovery 槽（recovery slot）** | 用 **recoveryWrapKey** 把同一個 `fileKey` AES-GCM 包裝；換機或未註冊裝置時走這槽。 |

## 2. Recovery wrapping key 的建立時機

在 **`setupRecoveryKey()`**（首次建立 vault）或使用 Recovery Key **`unlockWithRecoveryKey()`** 時：

1. Argon2id（`argon2id`）以 Recovery Key（UTF-8）與 **metadata 內建的 salt / memory / iterations** 運算。
2. 得到 **recoveryWrapKey**（長度為 `hash_length`，程式中為 32 bytes）。

參數常數請見 `lib/domain/recovery/kdf_descriptor.dart`。metadata 不包含 Recovery Key，只存放 KDF salt 與 hint（例如末四碼）。

## 3. 單一檔案的加密（`encryptBytes` / `encryptMarkdown`）

對每一個要落的檔案（日記、`manifest.json.enc`、附件 `.enc`）流程相同：

```
1. 產生隨機 fileKey（AES-256）
2. 呼叫 DeviceKeyManager.wrapWithDeviceKey → 裝置槽 ciphertext + nonce + slotId ...
3. 用 recoveryWrapKey 對同一 fileKey 做 AES-GCM → recovery 槽
4. 組 EncryptedDocumentHeader（JSON）：schema、cipher=aes-256-gcm、金鑰槽清單、內容 nonce ...
5. 將 header 序列化為 UTF-8 JSON 位元組，稱為 AAD **headerBytes**
6. 使用 fileKey + 內容 nonce，對明文做 AES-GCM encrypt，並把 MAC 附加在 ciphertext 末尾
7. 寫檔：**LDJ2** + **Big-Endian UInt32(header 長度)** + **headerBytes** + **ciphertext||tag**
```

重點：

- **正文加密演算法**：`aes-256-gcm`；標頭明文存在檔開頭（可被讀取的只有 metadata，不含明文日記）。
- **AAD（Additional Authenticated Data）** 為整段 `headerBytes`，因此竄改標頭會導致正文 MAC 驗證失敗。
- 每檔內文使用**自己的**隨機 nonce（12 bytes）；GCM tag 為 16 bytes，接在 ciphertext 之後。

## 4. LDJ2 檔案二進位版面

| 區段 | 長度 / 說明 |
|------|-------------|
| Magic | 4 bytes ASCII `LDJ2` |
| `header_len` | 4 bytes，big-endian，無號 32-bit |
| Header | `header_len` bytes，UTF-8 JSON |
| Body | 剩餘位元組 = `ciphertext || GCM tag` |

若 magic 非 `LDJ2` 或 schema 非實作所支援版本，parse 階段即拒絕（不支援舊格式）。

## 5. 標頭（JSON）裡與金鑰相關的欄位（概念）

- `cipher`：內文必須為 `aes-256-gcm`。
- `nonce`：內文 GCM 的 nonce（Base64）。
- `key_slots`：至少含一個 `slot_type: device` 與一個 `slot_type: recovery`。
  - **device**：`wrap_algorithm` 為 `android-keystore-aes-gcm`，並帶 `platform` 等由 `DeviceKeyManager` 填入的欄位。
  - **recovery**：`wrap_algorithm` 為 `aes-256-gcm`，並內嵌 **相同 recovery 用的** `kdf` 描述（與 `recovery.json` 一致），供驗證／除錯與未來工具使用。

## 6. 日記寫入與衍生資料（Repository 層）

`VaultRepository.saveEntry` 在加密成功後：

1. 以 **暫存檔 + rename** 方式原子寫入 `.md.enc`（避免半寫入）。
2. 更新 SQLite：條目、附件路徑、搜尋用預覽、`contentHash` 等。
3. 重寫加密的 **`manifest.json.enc`**（內為 JSON 統計資訊的密文）。

**真實資料來源**是加密檔；索引與 Manifest 可依加密檔與規則重建（見〈其他主要流程〉）。
